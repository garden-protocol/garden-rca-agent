## 1. Service Architecture Overview

### What Each Service Does

**evm-executor** (Rust, port 4525)
The executor is the solver-side on-chain action engine. It receives orders with actions (initiate/redeem/refund) from the solver engine via its HTTP API, stores them in an in-memory queue, and batches them for on-chain submission using multicall. It registers itself with the solver engine registry on startup. It uses a keystore file protected by a password obtained via a separate unlock server (port 4526). One `EvmExecutor` loop runs per chain.

**evm-watcher** (Go)
The watcher monitors HTLC contract events on EVM chains and writes state transitions to a shared PostgreSQL database. It supports three contract generations: `gardenhtlc` (V1), `gardenhtlcv2` (V2), and `gardenhtlcv3` (V3). It polls logs on a configurable interval, maintains a 50-block overlap to avoid missed events, updates swap initiate/redeem/refund records in the DB, and tracks confirmation counts for initiated swaps.

**evm-relay** (Rust, port 4426)
The relay is the user-facing swap coordination service. It handles:
- **Initiation**: Accepts user EIP-712 signatures and submits initiate transactions via per-chain `HTLCExecutor` instances.
- **Redemption**: Polls pending orders from an orders API and calls redeem on destination chains when the source swap is initiated and a secret is available.
- **Instant Refund**: Generates pre-signed refund transactions for Bitcoin-side instant refunds.
- **Bitcoin redemption**: Triggers Bitcoin HTLC redemption.

The relay uses Redis as a transaction queue and PostgreSQL (via `OrderbookProvider`) for order state.

**garden-contract-hub** (Solidity)
Contains the HTLC smart contracts. Three EVM variants exist:
- `HTLC.sol` — Standard EVM chains (uses `block.number`).
- `ArbHTLC.sol` — Arbitrum (uses `ArbSys.arbBlockNumber()` for L2 block counting).
- `NativeHTLC.sol` / `ArbNativeHTLC.sol` — Same as above but for native ETH instead of ERC-20.

### Order Lifecycle on EVM

```
[User]  ---(EIP-712 sig)---> [evm-relay /initiate]
                                      |
                          GardenActionRequest (HTLCAction::InitiateWithUserSignature)
                                      |
                              [HTLCRequestSender]
                                      |
                              Redis TxPool (queue)
                                      |
                            [HTLCExecutor.run()]
                            (dry-run -> execute batch)
                                      |
                         Multicall3 on-chain transaction
                                      |
                         [evm-watcher polls logs]
                         Initiated event -> UpdateSwapInitiate() -> DB
                                      |
                  [evm-relay RedeemerService polls pending orders]
                    (source_swap.initiate_tx_hash is set,
                     destination_swap.initiate_tx_hash is set,
                     neither redeemed nor refunded)
                                      |
                     [EvmUserRedeemer.redeem()] / [BitcoinUserRedeemer.redeem()]
                                      |
                         GardenActionRequest (HTLCAction::Redeem)
                                      |
                              [HTLCRequestSender.execute()]
                                      |
                              Redis TxPool (queue)
                                      |
                           [HTLCExecutor] -> on-chain redeem()
                                      |
                         [evm-watcher] Redeemed event -> UpdateSwapRedeem() -> DB
```

**Refund path**: After timelock expiry, any caller can invoke `refund(orderID)` on-chain. The watcher picks up `Refunded` event and calls `UpdateSwapRefund()`. Note: the relay's `RefunderService` is currently commented out; COBI handles instant refunds directly. The relay does still run `InstantRefundTxUpdaterService` for Bitcoin-side orders.

### How Services Interact

| Producer | Consumer | Channel |
|---|---|---|
| Solver Engine | evm-executor `/execute` HTTP | HTTP POST |
| evm-executor | Chain (EVM RPC) | Multicall3 batch tx |
| evm-watcher | PostgreSQL DB | GORM writes |
| evm-relay `/initiate` | evm-relay HTLCExecutor | Redis TxPool |
| evm-relay RedeemerService | PendingOrdersProvider API | HTTP GET |
| evm-relay HTLCExecutor | Chain (EVM RPC) | Direct tx submission |

## 3. Critical Functions

### HTLC Contract (Solidity)

**`HTLC._initiate(funder_, initiator_, redeemer_, timelock_, amount_, secretHash_)`** — `src/swap/HTLC.sol:296`
- Computes `orderID = sha256(abi.encode(chainid, secretHash_, initiator_, redeemer_, timelock_, amount_, address(this)))`.
- Reverts with `HTLC__DuplicateOrder` if `orders[orderID].timelock != 0`.
- Transfers tokens via `safeTransferFrom`.
- Emits `Initiated(orderID, secretHash_, amount_)`.

**`HTLC.redeem(orderID, secret)`** — `src/swap/HTLC.sol:234`
- Verifies `redeemer != address(0)` (order exists).
- Verifies `fulfilledAt == 0` (not already fulfilled).
- Recomputes `orderID` from `sha256(abi.encode(...))` to verify secret.
- Sets `fulfilledAt = block.number`.
- Emits `Redeemed(orderID, secretHash, secret)`.
- Transfers tokens to `redeemer`.

**`HTLC.refund(orderID)`** — `src/swap/HTLC.sol:266`
- Requires `timelock > 0` (order exists).
- Requires `fulfilledAt == 0`.
- Requires `initiatedAt + timelock < block.number` (expired).
- Sets `fulfilledAt = block.number`, emits `Refunded(orderID)`.
- Transfers tokens to `initiator`.

**`HTLC.instantRefund(orderID, signature)`** — `src/swap/HTLC.sol:331`
- Skips timelock check; requires redeemer's EIP-712 signature (or caller == redeemer).
- Sets `fulfilledAt` and emits `Refunded`.

**ArbHTLC difference**: Uses `ArbSys(0x0000000000000000000000000000000000000064).arbBlockNumber()` instead of `block.number` for `initiatedAt`, `fulfilledAt`, and timelock expiry check. This means timelocks on Arbitrum refer to L2 block numbers, not L1.

### evm-executor

**`EvmExecutor::run()`** — `executor/src/core/executor/evm.rs:41`
- Infinite loop calling `process_pending_orders()`, sleeping `polling_interval` ms between iterations.

**`EvmExecutor::process_pending_orders()`** — `executor/src/core/executor/evm.rs:55`
- Pops up to `BATCH_LIMIT = 64` orders from cache.
- Calls `order_executor.dry_run_and_filter()` — if this fails entirely, all orders fail.
- Calls `order_executor.submit_requests()` on passing orders.
- Updates each action's status in cache via `update_statuses()`.

**`GardenOrderExecutor::submit_transaction()`** — `executor/src/adapters/order_executor/mod.rs:158`
- Gets nonce from provider.
- Estimates EIP-1559 fees with `provider.estimate_eip1559_fees()`.
- On `replacement transaction underpriced` error, retries with `fees * 3` for both `max_fee_per_gas` and `max_priority_fee_per_gas`.
- Logs `"transaction submitted"` on success.

**`GardenOrderExecutor::submit_requests_internal()`** — `executor/src/adapters/order_executor/mod.rs:132`
- After submission, calls `handler.handle_tx()` to wait for confirmation.
- Returns `ExecutionStatus::Failed { reason: "transaction dropped or reverted" }` on `Reverted` or `Dropped`.
- Returns `ExecutionStatus::Success { tx_hash }` on `Confirmed`.

### evm-relay (HTLCExecutor)

**`HTLCExecutor::run()`** — `relay/src/executor/tx_executor.rs:72`
- Pops up to `batch_size = 128` requests from Redis.
- Calls `action_executor.actions_dry_run()` — on full failure, adds all requests back for retry.
- Filters into `valid_requests` and `rejected_requests`.
- Calls `action_executor.execute()` with a nonce from `NoncePool`.
- On success: records result, sends response channels, calls `pool.add_submitted_transaction()`.
- On failure: calls `pool.rollback(nonce)` and `add_requests_for_retry()`. If error contains `"nonce too low"` or `"nonce too high"`, calls `handle_nonce_error()` to sync nonce from chain.

**`HTLCExecutor::handle_nonce_error()`** — `relay/src/executor/tx_executor.rs:180`
- Fetches confirmed nonce from provider.
- Calls `pool.reset(confirmed_nonce)` to sync the `NoncePool`.

**`TransactionHandler::run()`** — `relay/src/executor/tx_handler.rs:84`
- Pops submitted transactions from Redis queue.
- Calls `pending_tx_handler.handle_tx()` (waits up to `CONFIRMATION_TIMEOUT = 30` seconds).
- On `Confirmed`: marks completed in Redis.
- On `Reverted`: adds requests back for retry.
- On `Dropped`: releases nonce via `pool.rollback()`, adds requests back for retry.

**`HTLCRequestSender::execute()`** — `relay/src/executor/mod.rs:168`
- Checks `submitted_transactions_count >= MAX_PENDING_REQUESTS (25)` and returns `MaxPendingRequestsReached`.
- Adds request to TxPool; does not wait for response.

**`HTLCRequestSender::execute_with_response()`** — `relay/src/executor/mod.rs:188`
- Same check as above.
- Waits up to `request_timeout_secs = 30` seconds for response via `oneshot` channel.
- Returns `InternalError("Request timed out")` on timeout.

**`RedeemerService::is_order_ready_for_redemption()`** — `relay/src/redeemer/mod.rs:173`
- Checks: `source_swap.initiate_tx_hash.is_some()` AND source not redeemed (unless destination is Bitcoin) AND source not refunded AND `destination_swap.initiate_tx_hash.is_some()` AND destination not redeemed/refunded.

**`EvmUserRedeemer::redeem()`** — `relay/src/redeemer/evm.rs:46`
- Validates secret: `sha256(secret_bytes) == expected_hash`, returns `"Secret hash mismatch"` on failure.
- Builds `GardenActionRequest { action: HTLCAction::Redeem { secret }, ... }`.
- Calls `executor_tx.execute()` (fire-and-forget, no response wait).

### evm-watcher

**`Watcher::pollEvents()`** — `watcher/pkg/watcher/watcher.go:252`
- Gets latest block, constrains range to `MaxBlockSpan`.
- Fetches from `fromBlock - 50` to handle reorgs/overlap (skipping already-processed event IDs via `processedEvents` cache).
- After log processing, calls `UpdateHTLCConfirmations()`.
- Persists current block to `BlockNumbersPersistor`.

**`Watcher::worker()`** — `watcher/pkg/watcher/watcher.go:213`
- Processes `EventJob` from queue.
- On `"no swap found"`: logs error, **does not retry**, skips.
- On `"failed to write matched order"`: sleeps 250ms and re-queues for retry.
- On success: marks event ID in `processedEvents` cache (1-hour TTL).

**`Store::UpdateSwapInitiate()`** — `watcher/pkg/store/store.go:54`
- Updates swap record matching `swap_id`, `timelock`, `redeemer`, `asset`, `chain`.
- If `RowsAffected == 0`, falls through to `UpdateMismatchedAmount()` to handle cases where the on-chain amount differs from the expected amount.

**`Store::UpdateMismatchedAmount()`** — `watcher/pkg/store/store.go:158`
- Determines action via `internal.DetermineMismatchedAmountAction()` based on `allowedMismatchPercent` and `supportedMismatchedAmountChains`.
- Actions: `MismatchedAmountUpdateSwaps`, `MismatchedAmountUpdateDestinationSwap`, `MismatchedAmountRefundSourceSwap`.

## 4. Known Failure Patterns

### Initiate Failures

**`HTLC__DuplicateOrder` (0x90c06174)**
- Condition: `_initiate()` called with same `(chainid, secretHash, initiator, redeemer, timelock, amount, contractAddr)` tuple.
- Cause: Duplicate initiation attempt; order already exists on-chain.
- Fatal for that request. The dry-run will catch this before submission.
- Log in executor: `"dry run failed"` (warn) with `action = Initiate`, error contains `DuplicateOrder`.

**`HTLC__ZeroAmount` (0x4d15835c)**
- Condition: `amount == 0` in initiate call.
- Seen in relay test: explicitly corrupting `init_orders[4].swap.amount = U256::ZERO` causes dry-run failure.
- Fatal for that request.

**`HTLC__InvalidInitiatorSignature` (0x3b3fd9ae)**
- Condition: EIP-712 signature verification fails in `initiateWithSignature()`.
- Cause: Wrong signer, wrong domain (chain ID mismatch), or malformed signature.
- Relay logs at warn level: `"Failed to extract EVM swap data"` or signature parse error.

**`replacement transaction underpriced`**
- Condition: Relay/executor submits a tx with same nonce as a pending tx but insufficient gas premium.
- Recovery: `GardenOrderExecutor.submit_transaction()` retries with `fees * 3` automatically.
- If retry also fails: `ExecutionStatus::Failed { reason: "..." }`.

**`nonce too low` / `nonce too high`**
- Condition: `NoncePool` is out of sync with chain state.
- Recovery: `HTLCExecutor.handle_nonce_error()` fetches confirmed nonce and calls `pool.reset()`.
- All affected requests go back into retry queue via `add_requests_for_retry()`.

**`Max pending requests reached`**
- Condition: `submitted_transactions_count >= MAX_PENDING_REQUESTS (25)` in relay.
- Response: HTTP 429 "Transaction queue is full. Please try again later."
- Transient: clears as transactions get confirmed.

**Dry run failure (relay executor)**
- If `action_executor.actions_dry_run()` fails entirely (RPC error), all requests go to `add_requests_for_retry()`.
- Log: `"Error simulating dry run: {:?}"` at error level.
- Individual request dry-run failures: `ExecutorError::InvalidSwapRequest(reason)`, request marked as `Rejected`.

**`Request timed out`**
- Condition: `execute_with_response()` waits 30 seconds for response, none arrives.
- Cause: Executor is overloaded, or Redis is slow.
- HTTP response: 500 "Internal error: Request timed out".

### Watcher Failures

**`"no swap found with the given orderID: <id>"`**
- Source: `store.UpdateSwapRedeem()` or `store.UpdateSwapRefund()` — `RowsAffected == 0`.
- Cause: Watcher received a `Redeemed`/`Refunded` event for an order not in the DB. Could be a reorg artifact, a manually-initiated order, or the order was created on a different environment.
- Not retried. Logged at error level: `"No swap found with the given details"`.

**`"failed to write matched order"`**
- Retried with 250ms sleep, re-queued indefinitely.
- Cause: Transient DB write failure.

**`"failed to get l1 block: ..."`**
- Arbitrum only: watcher calls `eth_getTransactionReceipt` to get `l1BlockNumber`.
- If RPC times out or returns error, `pollEvents()` returns error and skips that poll cycle.
- Logged at error: `"Error polling events"`.

**Job queue full (drop)**
- Condition: Worker pool backlog exceeds `QueueSize = 1000`.
- Behavior: Event silently dropped.
- Log at warn: `"Job queue is full, dropping event"` with contract address and event ID.
- This can cause missed initiations if the system is severely backlogged.

**Block persistence failure**
- `blockpersistor.Run()` logs `"failed to persist watcher block numbers"` and continues.
- If the service restarts before persistence, the watcher will re-scan from the last persisted block (or config `startBlock`), causing duplicate event processing.
- Duplicate events are handled by the `processedEvents` in-memory cache (1-hour TTL), but not across restarts.

### Redemption Failures

**`"Secret hash mismatch"` in EvmUserRedeemer**
- Condition: `sha256(secret_bytes) != order.create_order.secret_hash`.
- Fatal for that order; it goes to error log and is not retried (non-redeemable cache not set, so it will retry on next poll).
- Source: `relay/src/redeemer/evm.rs:33`.

**`"No redeemer available for chain"`**
- Condition: `RedeemerService` finds no `UserRedeemer` registered for the destination chain.
- Order is added to `non_redeemable_cache` (TTL 3600s) and skipped.
- Log at warn: `"No redeemer available for chain"`.

**`"Failed to send to executor: Max pending requests reached"`**
- Relay executor is at capacity (25 pending txs).
- The order is NOT cached as non-redeemable; it will be retried on the next 2-second poll cycle.

**`HTLC__OrderFulfilled` (0x356b842c)**
- On-chain redeem called on already-fulfilled order.
- Dry-run catches this; request marked `Rejected`.
- Cause: Double-redemption attempt, or both relay and another party redeemed simultaneously.

**`HTLC__IncorrectSecret` (0x3dbd7ab4)**
- The `sha256(secret)` does not match the committed secretHash in the orderID derivation.
- Dry-run catches this; fatal.

**Transaction dropped (relay `TransactionHandler`)**
- `HandleTxResult::Dropped`: nonce released via `rollback()`, requests added to retry.
- Log at error: `"Dropped transaction"`, then `"Dropped transaction added back to pool for retry, nonce released"`.

**Transaction reverted (relay `TransactionHandler`)**
- `HandleTxResult::Reverted`: requests added to retry (no nonce release — nonce already consumed).
- Log at error: `"Reverted transaction"`.

## 7. On-Chain Checks per Failure Type

### `missed_init` — Source or Destination HTLC Not Initiated

**Hypothesis**: The initiation transaction was never submitted, submitted but dropped, or submitted but reverted.

**On-chain checks**:
1. Call `orders(orderID)` on the HTLC contract. If `timelock == 0`, the order does not exist.
2. Check `initiate_tx_hash` in DB — if null, no initiation was recorded by watcher.
3. If `initiate_tx_hash` is set but order does not exist on-chain, the watcher may have processed a reorg'd block.
4. Check the transaction by hash: is it in a block? Did it revert? Revert reason is in the transaction receipt logs.
5. For relay: check Redis status of the `request_id` — is it `Pending`, `Submitted`, `Confirmed`, or `Rejected`?
6. For executor: check executor order cache status via `GET /status/{action_id}`.
7. If `Rejected`, look at the rejection reason — likely a dry-run failure.
8. If `Submitted` but never `Confirmed`, the transaction may be stuck (gas too low, nonce gap).

**Rules out**: If `orders(orderID).timelock > 0` on-chain, initiation succeeded — the issue is the watcher not processing the event.

---

### `deadline_approaching` — Timelock Near Expiry Without Destination Redemption

**Hypothesis**: Source is initiated, destination is initiated, but neither has been redeemed. The timelock is about to expire.

**On-chain checks**:
1. Call `orders(destOrderID)` on destination HTLC. Check `initiatedAt + timelock` vs current block number.
2. For Arbitrum: use `ArbSys(0x64).arbBlockNumber()` not `eth_blockNumber` for the correct comparison.
3. Check if `fulfilledAt == 0` (not yet redeemed or refunded).
4. Check relay logs for `"No redeemer available for chain"` or `"Skipping order redemption as secret is empty"`.
5. Verify the secret is available: check credentials provider API for the order's secret.
6. Check relay `RedeemerService` — is it running? Are there `"Failed to redeem order"` errors?
7. If the source has been redeemed but destination hasn't, the secret is revealed on-chain in the `Redeemed` event. Extract it from source chain logs and manually trigger redemption.

**Rules out**: If `orders(destOrderID).fulfilledAt > 0`, the order is already fulfilled.

---

### `stuck_tx` — Transaction Submitted But Not Confirmed

**Hypothesis**: A transaction is broadcast but stuck in mempool (gas too low, nonce gap, RPC issue).

**On-chain checks**:
1. Look up the `tx_hash` in a block explorer — is it pending, included, or unknown?
2. If pending: `max_fee_per_gas` may be below current base fee. Compare with `eth_getBlock('pending').baseFeePerGas`.
3. If unknown: it may have been dropped from mempool. Check if a transaction with the same nonce (from the same sender) was included — replacement tx.
4. For nonce gaps: call `eth_getTransactionCount(sender, 'latest')` vs `'pending'`. A gap indicates a dropped low-nonce tx is blocking higher nonces.
5. For relay: check `TxPool` Redis key for `submitted_transactions_queue` — is the tx still in the queue?
6. Check relay logs for `"Dropped transaction"` or `"Reverted transaction"`.
7. If the relay `TransactionHandler` timed out (30s CONFIRMATION_TIMEOUT), the tx was added back for retry with `add_requests_for_retry()`.

**Rules out**: If `eth_getTransactionReceipt(tx_hash)` returns a receipt with `status: 1`, the tx succeeded but the watcher may have missed the event.

---

### `redeem_failed` — Redemption Attempted But Failed

**Hypothesis**: Relay tried to redeem but the on-chain transaction reverted.

**On-chain checks**:
1. Find the failed redeem tx (from relay logs or `tx_hash` in Redis). Check the revert reason in the receipt.
2. `HTLC__OrderFulfilled (0x356b842c)`: Order already redeemed — another party redeemed first (race condition). This is fine; check if the secret was revealed.
3. `HTLC__IncorrectSecret (0x3dbd7ab4)`: Wrong secret submitted. Check the secret in the relay's credentials provider against the expected `secretHash` in `orders(orderID).secretHash` (derived from `orderID`).
4. `HTLC__OrderNotInitiated (0xca1eac12)`: The `redeemer` field is zero — order was never initiated.
5. Check `orders(orderID).fulfilledAt` — if non-zero, already fulfilled.
6. For dry-run failures: check relay logs for `"Invalid swap request: <reason>"` to understand why dry-run rejected the request.
7. Verify the relay's signer address has the correct role to call `redeem()` (any address can call redeem if they have the secret).

**Rules out**: If `orders(orderID).fulfilledAt > 0` AND `Redeemed` event exists with the correct `secret`, redemption succeeded.

---

### `refund_failed` — Refund Attempted But Failed

**Hypothesis**: Refund was attempted but reverted, or timelock has not yet expired.

**On-chain checks**:
1. Check `orders(orderID).initiatedAt + orders(orderID).timelock` vs current block.
   - For Arbitrum: compare against `ArbSys(0x64).arbBlockNumber()`.
   - For standard EVM: compare against `eth_blockNumber`.
2. `HTLC__OrderNotExpired (0x839f009c)`: Timelock has not expired yet. Wait.
3. `HTLC__OrderFulfilled (0xa7dc33de on ArbHTLC)` / `(0x356b842c on HTLC)`: Already redeemed or refunded.
4. `HTLC__OrderNotInitiated`: Order doesn't exist — wrong `orderID`, wrong contract address, or wrong chain.
5. For instant refund: check that the redeemer signature is valid. Call `instantRefundDigest(orderID)` and verify the signature against the redeemer address.
6. Check if `HTLC__InvalidRedeemerSignature (0xb41ed7a7 on HTLC)` was the revert reason — the instant refund signature is invalid or expired.
7. Verify the contract address being called matches the one the order was initiated on (check watcher DB for `asset` field or the original initiation event).

**Rules out**: If `orders(orderID).fulfilledAt > 0`, the order is done. Check whether it was `Redeemed` or `Refunded` by looking at which event was emitted.

---

### Arbitrum-Specific Considerations

The watcher translates Arbitrum L2 block numbers to L1 block numbers for V1 (`gardenhtlc`) contracts via `GetL1CurrentBlock()` (calls `eth_getTransactionReceipt` with `l1BlockNumber` field). For ArbHTLC (V3 on Arbitrum), `ArbSys.arbBlockNumber()` is used on-chain and the watcher uses the L2 block number directly.

When debugging timelock expiry on Arbitrum with ArbHTLC:
- The timelock comparison is `initiatedAt + timelock < ArbSys.arbBlockNumber()`.
- `initiatedAt` is the **L2** block number at the time of initiation.
- Do NOT use `eth_blockNumber` (may differ slightly from `ArbSys.arbBlockNumber()`); use `eth_call` to `ArbSys(0x64).arbBlockNumber()`.

---

### Mismatched Amount Events

If the watcher's `UpdateSwapInitiate()` returns `RowsAffected == 0` (order not found by exact amount match), it calls `UpdateMismatchedAmount()`. This handles cases where the user sent a slightly different amount. Depending on `allowedMismatchPercent` and whether the chain is in `supportedMismatchedAmountChains`:
- May update the source and destination swap amounts proportionally.
- May only update the destination swap.
- May flag the source swap for refund.

If `UpdateMismatchedAmount` also fails with `"no swap found"`, the watcher logs an error and the swap state is not updated — the order appears as uninitialized in the DB despite existing on-chain. This is a rare but known gap.

## 8. Shared Library Internals

The EVM chain services depend on two shared libraries:
- **blockchain** (Go) — used by `evm-watcher` for HTLC event parsing and contract bindings
- **garden-rs** (Rust) — used by `evm-executor` and `evm-relay` for HTLC contract interactions, multicall, fee resolution, and error handling

### 8.1 blockchain (Go) — EVM Package

**`evm/htlc.go`** — Go HTLC wallet and event client

| Interface | Methods | Used by |
|---|---|---|
| `HTLCWallet` | `Initiate()`, `Redeem()`, `Refund()`, `OrderID()` | Watcher (event parsing) |
| `HTLCClient` | `HTLCEvents(ctx, asset, fromBlock, toBlock)` | Watcher (log polling) |

**`Initiate()`**: Handles ERC20 approval check + initiate call. If `allowance < amount`, calls `erc20.Approve(swapper, MaxETHAmount)` first. Supports `InitiateWithSignature` (passes user EIP-712 sig to contract).

**`Redeem()`**: Calls `gardenhtlc.Redeem(orderID, secret)` via contract binding.

**`Refund()`**: Supports both normal refund (`gardenhtlc.Refund(orderID)`) and instant refund (`gardenhtlc.InstantRefund(orderID, sig)`) — routes based on whether `sig` is nil.

**`HTLCEvents()`**: Filters logs for `Initiated`, `Redeemed`, `Refunded` events from GardenHTLC contract. Parses into `HTLCInitiated{OrderID, SecretHash, Amount}`, `HTLCRedeemed{OrderID, Secret, SecretHash}`, `HTLCRefunded{OrderID}`.

**`evm/client.go`** — Multi-chain EVM client supporting Ethereum, Arbitrum, PolygonZK.

**`evm/bindings/contracts/htlc/gardenhtlc/GardenHTLC.go`** — Auto-generated Go bindings for the GardenHTLC Solidity contract. Contains `Initiate`, `InitiateWithSignature`, `Redeem`, `Refund`, `InstantRefund` methods and event parsers.

### 8.2 garden-rs (Rust) — EVM Crate

**`crates/evm/src/htlc/mod.rs`** — `GardenHTLC` struct

The main entry point for all EVM HTLC interactions from Rust services. Supports **6 contract variants**:

| Contract | Token Type | Version |
|---|---|---|
| `ERC20HTLC` | ERC-20 | V1 (`gardenhtlc`) |
| `ERC20HTLCv2` | ERC-20 | V2 (`gardenhtlcv2`) |
| `ERC20HTLCv3` | ERC-20 | V3 (`gardenhtlcv3`) |
| `NativeHTLC` | Native (ETH) | V1 |
| `NativeHTLCv2` | Native | V2 |
| `NativeHTLCv3` | Native | V3 |

Key features:
- **Caching**: `token_cache`, `htlc_cache`, `domain_cache`, `version_cache` — all `moka::Cache` with 10,000 capacity
- **GardenSolver mode**: When `is_garden_solver=true`, token allowance is disabled (solver contract handles approvals)
- **EIP-712 domain**: Cached per HTLC address for signature verification
- **Version auto-detection**: Queries contract to determine V1/V2/V3

**`crates/evm/src/htlc/v1.rs`, `v2.rs`, `v3.rs`** — Version-specific HTLC implementations:
- `get_calldata()`: Generates ABI-encoded calldata for initiate/redeem/refund actions

**`crates/evm/src/htlc/native.rs`, `nativev2.rs`, `nativev3.rs`** — Native token (ETH) HTLC implementations:

### 8.3 Multicall & Executor (garden-rs)

**`crates/evm/src/multicall.rs`** — `MulticallContract`

Batches multiple HTLC actions into a single Multicall3 transaction. Used by the evm-executor to execute initiations + redemptions in one tx.

`execute()` flow:
1. Builds `Multicall::builder(contract)` with all `CallParams`
2. Resolves fee options via `resolve_fee_options()` → either both `max_fee_per_gas` + `max_priority_fee_per_gas` provided, or estimates EIP-1559 fees
3. Optionally applies `gas_limit` and `nonce` from `TxOptions`
4. Submits and returns tx hash

Multicall3 contract addresses:
- Arbitrum: `0xA51c1fc2f0D1a1b8494Ed1FE312d7C3a78Ed91C0`
- Ethereum: `0x2279B7A0a67DB372996a5FaB50D91eAA73d2eBe6`

**`crates/evm/src/executor.rs`** — `GardenActionExecutor`

Orchestrates action execution:
1. `prepare_requests()`: validates swap params, parses asset address, looks up handler, generates calldata
2. `actions_dry_run()`: simulates without submitting (for pre-validation)
3. `execute()`: validates all requests, generates calldata, calls `executor.execute(calls, options)`

### 8.4 Fee Resolution (garden-rs)

**`crates/evm/src/common.rs:resolve_fee_options()`**

match (max_fee_per_gas, max_priority_fee_per_gas) {

### 8.8 Common RCA Scenarios Involving Shared Libraries

**Executor multicall fails silently for one action**:
The `GardenActionExecutor.execute()` validates all requests upfront and fails fast if any are invalid. Check `SimulationResult.error` from `actions_dry_run()`. Common causes: `"Invalid swap parameters"`, `"Invalid asset address"`, `"No handler found for action"`, `"Failed to generate calldata"`.

**Fee estimation failure on EVM**:
`resolve_fee_options()` calls `provider.estimate_eip1559_fees()` when no fee overrides given. If RPC is down, the error is sanitized by `sanitize_error()` → check for `"error sending request to RPC"` or `"RPC connection failed"` in executor logs.

**HTLC version mismatch**:
The executor auto-detects contract version. If it detects wrong version → `HTLCError::UnsupportedVersion{version, asset}`. This happens when a new contract is deployed but the executor cache still holds the old version (cache capacity 10,000 — rarely evicts).

**Native token initiate-with-signature rejected**:
`HTLCError::NativeTokenNotSupported` — instant error, no tx broadcast. Only ERC20 tokens support `InitiateWithSignature`.

**Allowance not set**:
Go watcher's `Initiate()` in `blockchain/evm/htlc.go` auto-approves `MaxETHAmount` if `allowance < amount`. Rust executor has `set_allowance` flag (default true, disabled for GardenSolver mode). If approval tx fails or times out (60s), initiation fails.

**Swap already redeemed/refunded**:
`HTLCError::AlreadyRedeemed` — executor tried to act on a completed swap. This is a race condition similar to Bitcoin's `"order fulfilled"` warn. The swap was completed by another party between the time the executor read the order and tried to act.
