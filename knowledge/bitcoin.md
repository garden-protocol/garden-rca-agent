## 1. Service Architecture Overview

### What Each Service Does

| Service | Repo | Role |
|---|---|---|
| **Executor** (cobi-v2) | Go | Polls orderbook every 2s, maps orders → HTLC actions, executes initiations/redemptions/refunds on-chain |
| **Cobi Watcher** (bit-ponder) | Go | Polls the indexer API for each HTLC address on a configurable interval, updates swap state in DB |
| **ZMQ Watcher** (bitcoin-watcher) | Rust | Subscribes to Bitcoin node ZMQ for raw mempool txs and new blocks; updates swap state in real time |
| **Relayer** (btc-relayer) | Rust | Polls pending orders every 2s, refunds expired Bitcoin HTLCs back to initiators |

### Order Lifecycle

```
1. Order matched → HTLC address derived from (initiator_pubkey, redeemer_pubkey, secret_hash, timelock)
2. Executor detects "initiate" action → broadcasts BTC to HTLC address
3. ZMQ Watcher picks up mempool tx → writes initiate_tx_hash + filled_amount to DB (unconfirmed)
   Cobi Watcher polls HTLC address → same but via indexer API (confirmation-based)
4. Both watchers track confirmations → when required_confirmations met, swap is "initiated"
5. Counterparty sees initiation → redeems on destination chain, revealing secret
6. Executor sees secret → submits redeem tx on Bitcoin HTLC
7. ZMQ Watcher / Cobi Watcher detect redeem spend → write redeem_tx_hash + secret to DB
8. REFUND PATH: if order > 24h old AND destination swap refunded (or never initiated)
   → Relayer submits refund tx spending the CSV timelock leaf
   → Watchers detect refund spend → write refund_tx_hash to DB
9. INSTANT REFUND: both parties co-sign to cancel cooperatively without waiting for CSV
```

### Service Interactions

- **Executor ↔ Orderbook**: `GetPendingOrders()` via HTTP, returns `MatchedOrder` objects
- **Executor ↔ Bitcoin node**: broadcasts txs via Bitcoin RPC/indexer
- **ZMQ Watcher ↔ Bitcoin node**: ZMQ subscription (`rawtx`, `rawblock` topics)
- **Both Watchers ↔ DB**: write `initiate_tx_hash`, `redeem_tx_hash`, `refund_tx_hash`, `secret`, `filled_amount`, confirmations
- **Relayer ↔ Orderbook**: `get_pending_orders()` via HTTP to find expired orders
- **Relayer ↔ Credentials API**: fetches initiator private key per order
- **All services ↔ Screener**: checks input addresses against blacklist before processing

## 3. Critical Functions

### Executor — Order Processing

**`executor/executor.go:Start` (line 131)**
Main ticker loop, fires every **2 seconds**. Also runs a cleanup ticker every 1 hour.

**`executor/executor.go:executeOrders` (line 178)**
1. `GetPendingOrders(fillerIDs...)` from orderbook
2. `filterWhitelisted(orders)` 
3. `GetLatestBlockNumbers()` — used for refund block calculations (failure here is non-fatal)
4. `mapToActions(orders, blockNumbers)` — maps each order to an `HTLCAction`
5. `filterActionsInCache()` — skip actions already in-flight
6. `splitActionsByChains()` — group by chain asset
7. `processActions(htlcActions)` in goroutines per chain asset

**`executor/executor.go:processActions` (line 233)**
- Timeout: **600 seconds** per batch
- Calls `chainWorker.Execute(ctx, actions...)`
- Calls `handleActionResponse()` per result

**`executor/executor.go:handleActionResponse` (line 262)**
- **Warns** (not errors) on: `nonce too low`, `order fulfilled`, `failed to wait for transaction`, `replacement transaction underpriced` — these mean the counterparty already acted, not a real failure
- **Errors** on all other failures
- On success: caches `action.ID() → txHash`

**`bitcoin/actions.go:Execute` (line 73)**
- Tries batch execution first
- On batch failure: falls back to individual execution per action
- Each individual failure is included in the response slice (not returned as error)

**`bitcoin/fee.go:FeeSuggestion` (line 19)**
- Iterates fee estimators in order, returns first successful result
- If all fail, returns error with last error message

### Cobi Watcher — Event Processing

**`watcher/watcher.go:scheduleJobs`**
- Fires on `cfg.Watcher.Interval` milliseconds
- Throttles uninitiated swaps by age (see Section 5)
- `jobQueue` buffer size: **500**
- Worker timeout per observable: **5 seconds**

**`watcher/watcher.go:processObservable` (line 302)**
1. Blacklist cache check (7-day TTL)
2. `HTLCClient.HTLCEvents()` — fetches all events for address
3. `removeDuplicates()` — dedup by tx hash
4. `eventCache.Has()` — skip if already processed
5. `filterEvents()` → init/redeem/refund slices
6. `handleInits()` → screens VIN addresses, updates `initiate_tx_hash`, `filled_amount`
7. `handleRedeems()` → updates `redeem_tx_hash`, `secret`
8. `handleRefundOrRedeem()` → may reclassify refund as redeem if output → `bitcoin_optional_recipient`
9. `HandleTimestamps()` → sets initiate/redeem/refund timestamps (async)
10. **Safety check**: if `init_tx_timestamp < swap.created_at` → blacklist and skip
11. `store.UpdateSwap()`

**`watcher/watcher.go:handleConfirmations`**
- Fires every **5 seconds**
- Calls `indexer.GetTipBlockHeight()` → `store.UpdateCurrentConfirmations()`

### ZMQ Watcher — Transaction Parsing

**`src/core/watcher/tx_event.rs:parse_swap_events`**
- Skips coinbase txs
- `process_deposits_in_tx()` — detects initiations from tx outputs
- `process_withdraws_in_tx()` — detects redeems/refunds from tx inputs

**`process_deposits_in_tx` (detect initiation)**
1. For each tx output, derive address from `script_pubkey`
2. Look up address in swap cache
3. **Amount must match exactly**: `swap.amount == output.value.to_sat()`
4. Amount must be within `processible_amount_limits.min..=max`
5. Screen VIN addresses via screener — if blacklisted, emit `is_blacklisted=true` event
6. If in mempool (block_height=0): `filled_amount = 0` (confirmed only)

**`process_withdraws_in_tx` (detect spend)**
1. For each tx input: extract taproot address from control block
2. Look up address in swap cache
3. `get_htlc_spend_type()`:
   - witness len **4**, `witness[0].len != witness[1].len` → **Redeem** (secret = `witness[1]`)
   - witness len **4**, `witness[0].len == witness[1].len` → **Instant Refund** (2-of-2 multisig)
   - witness len **3** → **Refund** (CSV timelock)

**`src/core/watcher/tx_processor.rs:TxIndexer.index`**
- Batch size: up to **256 txs** at a time
- Sleep **100ms** between batches
- `dedup_processed_events()`: TTL **4800s** (1h20m); re-check mempool after **3 min** buffer
- Reorg protection: if incoming block < stored block → verify tx is unconfirmed via indexer first

**`get_tx_with_retry`**
- Retries: **32 attempts**
- Initial delay: **256ms**, doubles each attempt, caps at **2s**
- TX cache TTL: **5 minutes**

### Relayer — Refund Logic

**`src/refunder/mod.rs:process_refund_orders` (line 79)**
1. `get_pending_orders(&chain, None)` 
2. `filter_orders()` — apply `validate_refund_order()` + cache dedup
3. `retrieve_credentials()` — fetch initiator private key
4. For each valid order: `refund_order()` → `BitcoinHTLC.refund(&htlc_params, &recipient)`
5. Failed orders → `cache.mark_orders_invalid()` (prevents retry)

**`src/validate.rs:validate_refund_order` (line 16)**
Fails fast on any of:
1. Source chain is not bitcoin
2. No `initiate_tx_hash`
3. `refund_tx_hash` already set
4. `redeem_tx_hash` already set (already redeemed)
5. Destination swap was initiated but not yet refunded
6. **Order newer than 24 hours** — most common reason for skip
7. No UTXOs found at HTLC address

## 4. Known Failure Patterns

### Pattern 1: Executor Cannot Initiate — Fee Too Low

**Symptom**: `Failed to execute action` in executor logs; tx not in mempool or dropped
**Root cause**: `FeeSuggestion()` returned a fee too low for network conditions, or all fee estimators failed
**Log signatures**:
- `executor/bitcoin/fee.go`: `"fee suggestion: <err>"` — all estimators failed
- `executor/executor.go`: `"Failed to execute action"` with fee-related error
- `executor/executor.go`: `"error processing actions"` 

**On-chain**: tx absent from mempool, or in mempool with very low sat/vbyte

### Pattern 2: Executor Skipped — Action Already In Cache

**Symptom**: Order stays pending but no new tx broadcast; no error logged
**Root cause**: `filterActionsInCache()` found the action already attempted; dedup preventing re-execution
**Log signatures**:
- `executor/executor.go`: `"Mapper: NoOpHTLCAction"` — order mapped to no-op
- No error, just silence

**On-chain**: previous tx may be stuck in mempool or confirmed at wrong amount

### Pattern 3: Cobi Watcher Misses Initiation

**Symptom**: `initiate_tx_hash` never written to DB despite confirmed on-chain tx
**Root cause candidates**:
- Swap not in watcher's observables (not yet in DB when watcher polled)
- Amount mismatch (partial fill not yet complete)
- Indexer API unavailable: `"failed to get HTLC events for <addr>"`
- Observable throttled: uninitiated swap > 30 min old → polled only every ~15 min
**Log signatures**:
- `watcher/watcher.go`: `"failed to get HTLC events for <addr>: <err>"`
- `watcher/watcher.go`: `"Error processing observable"`
- `watcher/watcher.go`: `"Failed to fetch and queue jobs"`

### Pattern 4: ZMQ Watcher Misses Initiation

**Symptom**: `initiate_tx_hash` missing; ZMQ watcher running but no event
**Root cause candidates**:
- Swap not yet in pending swap cache (poller interval not elapsed)
- Amount mismatch: `swap.amount != output.value.to_sat()` exactly
- Amount outside `processible_amount_limits`
- ZMQ connection dropped: `"Task exited unexpectedly"` in logs
**Log signatures**:
- `src/core/watcher/watcher.rs`: `"Task exited unexpectedly"` with task name
- `src/core/watcher/tx_event.rs`: no log on amount mismatch (silent skip)
- `src/adapters/zmq_events.rs`: ZMQ connection errors

### Pattern 5: Reorg Causes State Regression

**Symptom**: `initiate_tx_hash` reverts to empty after being set
**Root cause**: Bitcoin reorg — tx that was confirmed is now unconfirmed
**Behavior**: ZMQ watcher detects incoming_block < stored_block, verifies tx is unconfirmed via indexer, then calls `update_swaps(..., allow_degrade: true)`
**Log signatures** (bitcoin-watcher):
- Warning logs from `split_reorg_downgrade_events` when txid lookup fails

### Pattern 6: Refund Too Early — Relayer Skips

**Symptom**: Order appears stuck, relayer ignores it
**Root cause**: `validate_refund_order` skips orders newer than 24 hours OR destination swap not yet refunded
**Log signatures**:
- `btc-relayer/src/refunder/mod.rs`: `"No valid and unprocessed orders found, skipping refunder service"`
- `btc-relayer/src/validate.rs`: bail messages are not directly logged but propagated

### Pattern 7: Refund Fails — UTXO Spent or Fee Issue

**Symptom**: Relayer attempts refund, `"Failed to refund order"` in logs
**Root cause**: 
- UTXO already spent (user redeemed or double-spend)
- Fee estimator failure: all of mempool + blockstream APIs down, no fixed fallback
- CSV timelock not yet elapsed (order not old enough on-chain)
- Credentials retrieval failed

**Log signatures**:
- `btc-relayer/src/refunder/mod.rs`: `"Failed to refund order" order_id=<id> error=<err>`
- `btc-relayer/src/app.rs`: `"At least one fee rate estimator must be configured"` on startup

### Pattern 8: Blacklisted Order Silently Dropped

**Symptom**: No swap state updates; order frozen; no obvious error
**Root cause**: VIN address matched screener blacklist
**Behavior**:
- Cobi watcher: `MarkBlacklisted(swap_id)` in DB + adds to in-memory blacklist cache (7d TTL)
- ZMQ watcher: emits `SwapEvent { is_blacklisted: true }` → `mark_blacklisted()` in DB
**Log signatures**:
- `bit-ponder/watcher/watcher.go`: `"Blacklisted address found"` with swap_id
- `bitcoin-watcher/src/core/watcher/tx_event.rs`: `"TX has inputs from a blacklisted address"` with tx_id

### Pattern 9: Refund Misclassified as Redeem (Cobi Watcher)

**Symptom**: `refund_tx_hash` not set but `redeem_tx_hash` is, even though user refunded
**Root cause**: The refund tx sends output to `bitcoin_optional_recipient` — watcher reclassifies it as a redeem in `handleRefundOrRedeem()`
**When it happens**: BTC→BTC orders where destination is also Bitcoin; user provides `bitcoin_optional_recipient` in `additional_data`

### Pattern 10: Executor Error Swallowed — "Warned" Errors

**Symptom**: Action silently skipped; no retry; no error alert
**Root cause**: `handleActionResponse` downgrades to WARN for specific errors
**Affected errors**: `nonce too low`, `order fulfilled`, `failed to wait for transaction`, `replacement transaction underpriced`
**Action**: These are expected race conditions; the executor will pick up the correct state on next tick

## 7. On-Chain Checks per Failure Type

### `missed_init` — Expected initiation not found

1. Derive HTLC address from `(initiator_pubkey, redeemer_pubkey, secret_hash, timelock)` using the P2TR tapscript
2. Check UTXOs at the HTLC address — is the expected amount present?
3. If UTXO present but `initiate_tx_hash` not in DB: watcher missed it. Check if amount matches exactly (cobi watcher requires exact match; ZMQ watcher also requires exact match + within processible limits)
4. Check mempool: is the funding tx pending with low fee? (`get_mempool_entry(txid)`)
5. Check executor logs for `"Executed action"` + txHash for this order
6. Check if executor action was filtered by cache: `"Mapper: NoOpHTLCAction"` 
7. Verify required confirmations: `current_confirmations >= required_confirmations`

### `deadline_approaching` — Order about to expire

1. Check `initiate_tx_hash` in DB — was the swap initiated?
2. If initiated: check confirmations. Is it below `required_confirmations`?
3. Check mempool: is the initiation tx stuck? What is its fee rate vs. current network?
4. Check if counterparty redeemed on destination chain (secret revealed?)
5. Check relayer: will it attempt refund? Order must be > 24h old
6. Verify CSV timelock: has `timelock` blocks elapsed since initiation block?
7. Check if executor has a pending refund action for this order

### `stuck_order` — Order neither progressing nor failing

1. Check `initiate_tx_hash`, `redeem_tx_hash`, `refund_tx_hash` in DB — what state is it in?
2. If `initiate_tx_hash` set but `redeem_tx_hash` not: check if counterparty's destination swap has been redeemed (secret revealed). If yes, executor should be acting — check executor logs for this order's createID
3. Check executor cache: is action in `filterActionsInCache`? If stuck there, the tx may be confirmed but cache not cleared
4. Check if order is blacklisted: query DB for blacklist status; check screener for VIN addresses
5. For refund path: check relayer `validate_refund_order` conditions — order must be > 24h, destination refunded, no existing refund_tx_hash

### `failed_redeem` — Redemption transaction failed or missing

1. Check if secret is available: look for `redeem_tx_hash` on destination chain
2. Check `redeem_tx_hash` in DB — was the ZMQ or cobi watcher event written?
3. Check if secret was extracted: look for `secret` field in DB swap record
4. Check executor logs for `"Action execution skipped"` (order fulfilled / nonce too low) — executor may have seen it as a race condition
5. Verify the redeem tx witness: 4-element witness where `witness[0].len != witness[1].len` (otherwise mistakenly classified as instant refund)
6. Check if HTLC UTXO is already spent: if yes by whom and to what address?

### `failed_refund` — Refund transaction failed or not attempted

1. Verify relayer `validate_refund_order` conditions:
   - Is `initiate_tx_hash` set?
   - Is order > 24 hours old?
   - Is destination swap refunded (if it was ever initiated)?
   - Are UTXOs still present at HTLC address?
2. Check relayer logs for `"Failed to refund order"` with error details
3. Check if CSV timelock elapsed: `current_block_height - initiate_block_number >= timelock`
4. Check fee estimator: are mempool/blockstream APIs reachable? Is `fixed_fee` configured as fallback?
5. Verify HTLC address still has UTXOs — if empty, check who spent it
6. If UTXO spent to initiator: refund already succeeded but `refund_tx_hash` not written → watcher missed the spend event

### `blacklisted_order` — Order frozen due to screener

1. Check DB for blacklist flag on the swap
2. Check screener for VIN addresses from the initiation transaction
3. Check watcher logs for `"Blacklisted address found"` or `"TX has inputs from a blacklisted address"`
4. Verify screener service is reachable (both watchers fail-open on screener errors, defaulting to non-blacklisted)
