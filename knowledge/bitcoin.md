## 1. Service Architecture Overview

### What Each Service Does

| Service | Repo | Language | Role |
|---|---|---|---|
| **Executor** (cobi-v2) | cobi-v2 | Go | Polls orderbook every 2s, maps orders to HTLC actions, executes initiations/redemptions/refunds on-chain via `blockchain` lib |
| **Cobi Watcher** (bit-ponder) | bit-ponder | Go | Polls indexer API for each HTLC address on a configurable interval, updates swap state in DB. Uses `blockchain` lib for HTLC event parsing |
| **ZMQ Watcher** (bitcoin-watcher) | bitcoin-watcher | Rust | Subscribes to Bitcoin node ZMQ for raw mempool txs and new blocks; updates swap state in real time. Uses `garden-rs` for HTLC address derivation |
| **Relayer** (btc-relayer) | btc-relayer | Rust | Polls pending orders every 2s, refunds expired Bitcoin HTLCs back to initiators. Uses `garden-rs` for HTLC refund tx construction |
| **blockchain lib** | blockchain | Go | Shared library: Tapscript HTLC construction, tx batching (RBF/CPFP), fee estimation, indexer client, wallet operations |
| **garden-rs lib** | garden-rs | Rust | Shared library: Taproot HTLC construction, fee providers, orderbook primitives, tx signing |

### Shared Library Dependency Map

```
cobi-v2 (executor)     ──imports──> blockchain (Go)     ─── btc/htlc.go, btc/batcher.go, btc/fee.go
bit-ponder (watcher)   ──imports──> blockchain (Go)     ─── btc/event.go, btc/scripts.go, btc/indexer.go
bitcoin-watcher (zmq)  ──imports──> garden-rs (Rust)    ─── crates/bitcoin/src/htlc/, crates/primitives/
btc-relayer            ──imports──> garden-rs (Rust)    ─── crates/bitcoin/src/htlc/, crates/orderbook/
```

### Order Lifecycle

```
1. Order matched → HTLC address derived from (initiator_pubkey, redeemer_pubkey, secret_hash, timelock)
   - Go services: blockchain/btc/htlc.go:Address() → builds 3-leaf Tapscript tree → P2TR address
   - Rust services: garden-rs/crates/bitcoin/src/htlc/htlc.rs:get_htlc_address() → same derivation
   - Internal key: GardenNUMS = H + SHA256("GardenHTLC")*G (BIP-341 NUMS point)
2. Executor detects "initiate" action → broadcasts BTC to HTLC address via BatcherWallet
3. ZMQ Watcher picks up mempool tx → writes initiate_tx_hash + filled_amount to DB (unconfirmed)
   Cobi Watcher polls HTLC address → same but via indexer API (confirmation-based)
4. Both watchers track confirmations → when required_confirmations met, swap is "initiated"
5. Counterparty sees initiation → redeems on destination chain, revealing secret
6. Executor sees secret → submits redeem tx on Bitcoin HTLC (spends redeem leaf)
7. ZMQ Watcher / Cobi Watcher detect redeem spend → write redeem_tx_hash + secret to DB
8. REFUND PATH: if order > 24h old AND destination swap refunded (or never initiated)
   → Relayer submits refund tx spending the CSV timelock leaf
   → Watchers detect refund spend → write refund_tx_hash to DB
9. INSTANT REFUND: both parties co-sign SACP tx (SigHashSingle|AnyoneCanPay) to cancel cooperatively
```

### Service Interactions

- **Executor ↔ Orderbook**: `GetPendingOrders()` via HTTP, returns `MatchedOrder` objects
- **Executor ↔ Bitcoin node**: broadcasts txs via `blockchain/btc/indexer.go:electrsIndexerClient` (Electrs API)
- **Executor ↔ Batcher**: uses `blockchain/btc/batcher.go:BatcherWallet` for RBF/CPFP batching
- **ZMQ Watcher ↔ Bitcoin node**: ZMQ subscription (`rawtx`, `rawblock` topics)
- **Both Watchers ↔ DB**: write `initiate_tx_hash`, `redeem_tx_hash`, `refund_tx_hash`, `secret`, `filled_amount`, confirmations
- **Relayer ↔ Orderbook**: `get_pending_orders()` via HTTP to find expired orders (uses `garden-rs/crates/orderbook`)
- **Relayer ↔ Credentials API**: fetches initiator private key per order
- **All services ↔ Screener**: checks input addresses against blacklist before processing

## 3. Critical Functions

### HTLC Construction (blockchain lib — Go)

**`btc/scripts.go:GardenNUMS()` (line 329)**
Derives the internal key for all Garden Bitcoin HTLCs:
1. `r = SHA256("GardenHTLC")`
2. `H` = BIP-341 NUMS point (`0x50929b74c1a04954...`)
3. `rG = r * G` (secp256k1 generator)
4. `NUMS = H + rG`
This key is provably unspendable via the key path — all spends go through script paths.

**`btc/scripts.go:RedeemLeaf()` (line 34)**
```
OP_SHA256 <secret_hash> OP_EQUALVERIFY <redeemer_pubkey> OP_CHECKSIG
```
Requires: 32-byte secret whose SHA256 matches, plus redeemer's Schnorr signature.

**`btc/scripts.go:RefundLeaf()` (line 52)**
```
<timelock> OP_CHECKSEQUENCEVERIFY OP_DROP <initiator_pubkey> OP_CHECKSIG
```
Requires: CSV timelock elapsed (relative to UTXO confirmation), plus initiator's Schnorr signature. Timelock capped at `math.MaxUint16` (65535 blocks).

**`btc/scripts.go:MultiSigLeaf()` (line 73)**
```
<initiator_pubkey> OP_CHECKSIG <redeemer_pubkey> OP_CHECKSIGADD OP_2 OP_NUMEQUAL
```
Requires: both initiator and redeemer Schnorr signatures (2-of-2 via CHECKSIGADD).

**`btc/htlc.go:htlcLeaves()` (line 474)**
Constructs all 3 leaves → assembles Taproot script tree → computes output key. Leaf order: [redeem, refund, instantRefund]. Leaf version: `0xc0` (Tapscript v1).

**`btc/htlc.go:Address()` (line 124)**
Derives the P2TR address: `AssembleTaprootScriptTree(leaves...) → TapHash → ComputeTaprootOutputKey(internalKey, rootHash) → NewAddressTaproot()`.

**`btc/htlc.go:Execute()` (line 289)**
Batch execution: aggregates all sends (initiations), spends (redeems/refunds), and SACPs (instant refunds) into a single `wallet.Send()` call.

**`btc/htlc.go:canRefund()` (line 450)**
Checks if UTXO can be refunded: `utxo.BlockHeight + timelock - 1 <= currentTip`. If unconfirmed, needs `timelock + 1` more blocks.

### HTLC Construction (garden-rs — Rust)

**`crates/bitcoin/src/htlc/htlc.rs:GARDEN_NUMS`**
Same derivation as Go: `H + SHA256("GardenHTLC")*G`. Stored as `Lazy<XOnlyPublicKey>`. Produces identical addresses to the Go implementation.

**`crates/bitcoin/src/htlc/htlc.rs:construct_taproot_spend_info()`**
Builds Taproot tree with weights: redeem=1, refund=2, instant_refund=2 (Huffman-optimized for redeem being most common). This differs from Go which uses equal weights via `AssembleTaprootScriptTree`.

**`crates/bitcoin/src/htlc/htlc.rs:refund()`**
Builds refund tx: fetches UTXOs → creates inputs with CSV sequence → estimates fee → adjusts outputs → generates sighashes via `TapScriptSpendSigHashGenerator` → signs with Schnorr.

**`crates/bitcoin/src/htlc/primitives.rs:HTLCParams`**
```rust
pub struct HTLCParams {
    pub initiator_pubkey: XOnlyPublicKey,
    pub redeemer_pubkey: XOnlyPublicKey,
    pub amount: u64,
    pub secret_hash: [u8; 32],
    pub timelock: u64,
}
```
Converts from `SingleSwap` via `TryFrom` — parses hex pubkeys, decodes secret_hash, parses amount string.

### Transaction Batching (blockchain lib — Go)

**`btc/batcher.go:BatcherWallet`**
A wallet service that batches multiple HTLC actions into single transactions. Strategies:
- **RBF**: Replace-by-fee — each new batch replaces the previous unconfirmed tx
- **CPFP**: Child-pays-for-parent — new tx spends change output to bump fee of parent
Default: RBF strategy, PTI = 1 minute, FeeLevel = High.

**`btc/batcher.go:processBatch()` (line 373)**
Core loop: tries `createBatch()` first. If no new requests (`ErrBatchParametersNotMet`), tries `updateBatchFeeRate()` instead. This ensures stuck txs get fee-bumped even without new requests.

**`btc/rbf.go:createRBFBatch()` (line 20)**
1. Read pending requests from cache
2. Read latest RBF batch — if confirmed, create fresh batch; if unconfirmed, `reSubmitBatchWithNewRequests()`
3. RBF requires fee rate at least `currentFeeRate + 10` sat/vB

**`btc/rbf.go:createRBFTx()` (line 302)**
Recursive tx builder (max depth=2):
1. Fetch UTXOs for spend requests
2. Build transaction with inputs/outputs
3. Sign spends and sends
4. Calculate true vsize → estimate fee
5. If `newFeeEstimate > fee`: fetch more cover UTXOs if needed, recurse with updated fee

**`btc/cpfp.go:buildCPFPTx()` (line 228)**
Similar recursive approach (max depth=1):
1. Gather UTXOs, build tx, sign
2. Calculate true size, add buffer fee for signature sizes
3. If `newFeeEstimate > fee + feeOverhead`: recurse with higher fee

### Fee Estimation (blockchain lib — Go)

**`btc/fee.go:MempoolFeeEstimator`**
Fetches from `mempool.space/api/v1/fees/recommended`. Caches with configurable TTL. Returns `FeeSuggestion{Minimum, Economy, Low, Medium, High}`.

**`btc/fee.go:BlockstreamFeeEstimator`**
Fetches from `blockstream.info/api/fee-estimates`. Maps confirmation targets: 504→Minimum, 144→Economy, 6→Low, 3→Medium, 1→High.

**`btc/fee.go:EstimateVirtualSize()` (line 51)**
`baseSize + (swSize+3)/4` — standard segwit vsize calculation.

### Fee Estimation (garden-rs — Rust)

**`crates/bitcoin/src/fee_providers/multi.rs`**
`MultiFeeEstimator`: tries fee providers in order, returns first success. Same pattern as Go's `MultiFeeEstimator`.

**`crates/bitcoin/src/fee_providers/mempool.rs`**
Fetches from mempool.space API. `FeeLevel` enum: `Economy`, `Low`, `Medium`, `High`.

**`crates/bitcoin/src/batcher/fee.rs`**
`estimate_fee()`: calculates fee from input/output sizes and fee rate. `adjust_outputs_for_fee()`: deducts fee from outputs.

### Executor — Order Processing

**`executor/executor.go:Start` (line 131)**
Main ticker loop, fires every **2 seconds**. Also runs a cleanup ticker every 1 hour.

**`executor/executor.go:executeOrders` (line 178)**

**`executor/executor.go:processActions` (line 233)**
- Timeout: **600 seconds** per batch
- Calls `chainWorker.Execute(ctx, actions...)`
- Calls `handleActionResponse()` per result

**`executor/executor.go:handleActionResponse` (line 262)**
- **Warns** (not errors) on: `nonce too low`, `order fulfilled`, `failed to wait for transaction`, `replacement transaction underpriced`

- On success: caches `action.ID() → txHash`

**`bitcoin/actions.go:Execute` (line 73)**
- Tries batch execution first (all actions in one tx via `htlcWallet.Execute()`)
- On batch failure: falls back to individual execution per action
- Each individual failure is included in the response slice (not returned as error)

### Cobi Watcher — Event Processing

**`watcher/watcher.go:processObservable` (line 302)**
1. Blacklist cache check (7-day TTL)
2. `HTLCClient.HTLCEvents()` — uses `blockchain/btc/event.go` which parses witness scripts via `IsRedeemLeaf()`, `IsRefundLeaf()`, `IsMultiSigLeaf()`
3. `removeDuplicates()` → `eventCache.Has()` → `filterEvents()` → init/redeem/refund slices
4. `handleInits()` → screens VIN addresses, updates `initiate_tx_hash`, `filled_amount`
5. `handleRedeems()` → updates `redeem_tx_hash`, `secret`
6. `handleRefundOrRedeem()` → may reclassify refund as redeem if output → `bitcoin_optional_recipient`
7. **Safety check**: if `init_tx_timestamp < swap.created_at` → blacklist and skip

### ZMQ Watcher — Transaction Parsing

**`src/core/watcher/tx_event.rs:parse_swap_events`**
- `process_deposits_in_tx()` — detects initiations from tx outputs. Amount must match exactly: `swap.amount == output.value.to_sat()`. Must be within `processible_amount_limits.min..=max`
- `process_withdraws_in_tx()` — detects redeems/refunds from tx inputs. Uses witness analysis:
  - witness len **4**, `witness[0].len != witness[1].len` → **Redeem** (secret = `witness[1]`)
  - witness len **4**, `witness[0].len == witness[1].len` → **Instant Refund** (2-of-2 multisig)
  - witness len **3** → **Refund** (CSV timelock)

## 4. Known Failure Patterns

### Pattern 1: Executor Cannot Initiate — Fee Too Low or Estimator Failure

**Symptom**: `Failed to execute action` in executor logs; tx not in mempool or dropped
**Root cause**: All fee estimators failed, or returned fee too low for network conditions
**Library path**: `blockchain/btc/fee.go` → `MempoolFeeEstimator.FeeSuggestion()` / `BlockstreamFeeEstimator.FeeSuggestion()` → HTTP failures to mempool.space or blockstream.info APIs
**Log signatures**:
- `"fee suggestion: <err>"` — all estimators failed
- `"Failed to execute action"` with fee-related error
- `"error processing actions"`

### Pattern 2: Batcher Tx Stuck — RBF/CPFP Depth Exceeded

**Symptom**: Tx in mempool but not confirming; no fee bump happening
**Root cause**: `buildCPFPTx()` or `createRBFTx()` hit max recursion depth trying to estimate fees
**Library errors** (`blockchain/btc/batcher.go`):
- `ErrBuildCPFPDepthExceeded` — CPFP tx builder exceeded recursion limit (default depth=1)
- `ErrBuildRBFDepthExceeded` — RBF tx builder exceeded recursion limit (default depth=2)
- `ErrHighFeeEstimate` — estimated fee exceeds `TxOptions.MaxFeeRate`
- `ErrFeeDeltaHigh` — fee delta between current and required exceeds `TxOptions.MaxFeeDelta`
**Log signatures**: `"rebuilding CPFP transaction"` / `"rebuilding rbf tx"` with depth, fee, feeOverhead

### Pattern 3: Batcher Insufficient Funds

**Symptom**: Batch creation fails; new requests pile up
**Library errors** (`blockchain/btc/batcher.go`):
- `ErrInsufficientFundsInRequest` — `"have :<X>, need at least :<Y>"`. Wallet balance + script UTXOs < total sends + minimum 1000 sats
- `ErrNoFundsToSpend` — spend request scripts have zero UTXOs
- `ErrUnconfirmedUTXO` — RBF strategy requires confirmed UTXOs for spend requests
**Log signatures**: `"failed to create batch"` with underlying error

### Pattern 4: Indexer Submit Fails — Mempool Conflict

**Symptom**: Tx built and signed but rejected by network
**Library errors** (`blockchain/btc/indexer.go:SubmitTx`):
- `ErrAlreadyInChain` — `"transaction already in block chain"` (NoRetryError)
- `ErrTxInputsMissingOrSpent` — `"bad-txns-inputs-missingorspent"` (NoRetryError) — UTXO already consumed
- `ErrMempoolConflict` — `"txn-mempool-conflict"` (NoRetryError) — another tx spending same inputs
These are all wrapped in `NoRetryError` — the indexer client will NOT retry them.

### Pattern 5: HTLC Address Mismatch

**Symptom**: Funds sent to wrong address; watchers never detect initiation
**Root cause**: Mismatch in HTLC construction between Go and Rust implementations. Both use `GardenNUMS` as internal key, but leaf weights differ:
- Go (`blockchain`): uses `AssembleTaprootScriptTree(redeem, refund, instantRefund)` — equal weights
- Rust (`garden-rs`): uses `TaprootBuilder` with `REDEEM_LEAF_WEIGHT=1`, `OTHER_LEAF_WEIGHT=2` — Huffman-optimized
**IMPORTANT**: Despite different tree construction, the addresses MUST match. If they don't, it's a critical bug. Verify by deriving the address from both implementations with the same params.

### Pattern 6: Secret Validation Failure on Redeem

**Symptom**: `ErrInvalidSecret` when executor tries to redeem
**Library path**: `blockchain/btc/htlc.go:isSecretValid()` — `SHA256(secret) != htlc.SecretHash`
**Root cause**: Secret from destination chain doesn't hash to the expected secret_hash. Could be encoding issue (hex vs raw bytes) or wrong secret extracted from destination chain event.

### Pattern 7: Refund Too Early — Timelock Not Elapsed

**Symptom**: Relayer refund fails with block count error
**Library error**: `ErrHTLCNeedMoreBlocks(blocks)` — `"need more X blocks to refund"`
**Path**: `blockchain/btc/htlc.go:canRefund()` — checks `utxo.BlockHeight + timelock - 1 > currentTip`
**garden-rs equivalent**: `crates/bitcoin/src/htlc/htlc.rs:refund()` sets `Sequence(htlc_params.timelock)` — Bitcoin consensus enforces CSV

### Pattern 8: Executor Skipped — Action Already In Cache

**Symptom**: Order stays pending but no new tx broadcast; no error logged
**Root cause**: `filterActionsInCache()` found the action already attempted
**Log signatures**: `"Mapper: NoOpHTLCAction"` — order mapped to no-op

### Pattern 9: Cobi Watcher Misses Initiation

**Symptom**: `initiate_tx_hash` never written to DB despite confirmed on-chain tx
**Root cause candidates**:
- Swap not in watcher's observables
- Amount mismatch (watcher requires exact match)
- Indexer API unavailable
- Observable throttled: uninitiated swap > 30 min old → polled only every ~15 min

### Pattern 10: ZMQ Watcher Misses Initiation

**Symptom**: `initiate_tx_hash` missing; ZMQ watcher running but no event
**Root cause candidates**:
- Swap not yet in pending swap cache
- Amount mismatch: `swap.amount != output.value.to_sat()` exactly
- Amount outside `processible_amount_limits`
- ZMQ connection dropped

### Pattern 11: Instant Refund SACP Validation Failure

**Symptom**: Instant refund fails during SACP validation
**Library errors** (`blockchain/btc/htlc.go:validateInstantRefundSACP`):
- `ErrSACPInvalidInputsLen` — signed inputs count != UTXO count
- `ErrSACPInvalidInput` — input txid doesn't match UTXO
- `ErrSACPInvalidOutput` — output script doesn't match wallet address
- `ErrInvalidInstantRefundSACPWitnessLen` — witness must have exactly 4 elements, sigs must be 65 bytes
- `ErrInvalidInstantRefundScript` — script in witness doesn't match instant refund leaf
- `ErrInvalidControlBlock` — control block doesn't match expected

### Pattern 12: Dust Amount Rejection

**Symptom**: Transaction rejected or build fails
**Library constant**: `DustAmount = 546` satoshis (`blockchain/btc/btc.go`). Any output below this is rejected by miners. The batcher skips UTXOs with `Amount < DustAmount` when selecting cover UTXOs.
**Related error**: `ErrAmountLessThanDust` in wallet.go

### Pattern 13: Blacklisted Order Silently Dropped

**Symptom**: No swap state updates; order frozen
**Behavior**: VIN address matched screener blacklist
- Cobi watcher: `MarkBlacklisted(swap_id)` + 7-day cache
- ZMQ watcher: emits `SwapEvent { is_blacklisted: true }` → `mark_blacklisted()` in DB

### Pattern 14: Reorg Causes State Regression

**Symptom**: `initiate_tx_hash` reverts to empty after being set
**Root cause**: Bitcoin reorg — tx confirmed then unconfirmed
**Behavior**: ZMQ watcher detects incoming_block < stored_block, verifies tx is unconfirmed via indexer

## 7. On-Chain Checks per Failure Type

### `missed_init` — Expected initiation not found

1. Derive HTLC address from `(initiator_pubkey, redeemer_pubkey, secret_hash, timelock)` using `blockchain/btc/htlc.go:Address()` or `garden-rs:get_htlc_address()` — both produce P2TR address using GardenNUMS internal key
2. Check UTXOs at the HTLC address — is the expected amount present?
3. If UTXO present but `initiate_tx_hash` not in DB: watcher missed it. Check if amount matches exactly (both watchers require exact match)
4. Check mempool: is the funding tx pending with low fee?
5. Check executor logs for `"Executed action"` + txHash for this order
6. Check batcher logs: `"rebuilding rbf tx"` or `"failed to create batch"` with error
7. Check if batcher hit `ErrInsufficientFundsInRequest` — wallet may need funding

### `deadline_approaching` — Order about to expire

1. Check `initiate_tx_hash` — was the swap initiated?
2. If initiated: check confirmations. Below `required_confirmations`?
3. Check mempool: is the initiation tx stuck? Compare its fee rate vs current network (`FeeSuggestion().High`)
4. Check batcher: is fee bump happening? Look for `"batch fee updated"` or `"fee update skipped"`
5. Check if batcher hit `ErrHighFeeEstimate` or `ErrFeeDeltaHigh` — fee bump blocked by limits
6. Verify CSV timelock: has `timelock` blocks elapsed since initiation block?

### `stuck_order` — Order neither progressing nor failing

1. Check DB state: `initiate_tx_hash`, `redeem_tx_hash`, `refund_tx_hash`
2. If initiated but no redeem: check if counterparty redeemed on destination chain (secret revealed?). If yes, executor should be acting — check for `"Mapper: action"` with this order's createID
3. Check executor cache: is action stuck in `filterActionsInCache`?
4. Check batcher state: is there a pending batch? Is it stuck in mempool?
5. For refund path: order must be > 24h, destination refunded, no existing refund_tx_hash

### `failed_redeem` — Redemption transaction failed or missing

1. Check if secret is available from destination chain
2. Verify secret validity: `SHA256(secret) == secret_hash` (mirrors `blockchain/btc/htlc.go:isSecretValid()`)
3. Check executor logs for `"Action execution skipped"` (race condition) or `"Failed to execute action"`
4. Check batcher: did RBF/CPFP tx get rejected? Look for `ErrTxInputsMissingOrSpent` (UTXO already spent)
5. Verify witness classification: 4-element witness where `witness[0].len != witness[1].len` = Redeem

### `failed_refund` — Refund transaction failed or not attempted

1. Verify relayer conditions: `initiate_tx_hash` set, order > 24h, destination refunded, UTXOs present
2. Check `canRefund()`: `utxo.BlockHeight + timelock - 1 <= currentTip`. If not, `ErrHTLCNeedMoreBlocks`
3. Check relayer logs for `"Failed to refund order"` with error details
4. Check fee estimators: are mempool.space/blockstream APIs reachable?
5. Check if `garden-rs` refund tx builder failed: sighash generation, signing errors
6. If UTXO spent: who spent it? Derive the spending type from witness structure

### `blacklisted_order` — Order frozen due to screener

1. Check DB for blacklist flag
2. Check screener for VIN addresses from initiation tx
3. Check watcher logs for `"Blacklisted address found"` / `"TX has inputs from a blacklisted address"`
4. Both watchers fail-open on screener errors (default to non-blacklisted)

## 9. Investigation Runbooks

### 9.1 DestInitPending Investigation (Bitcoin as destination)
1. **Solver-engine**: Was order mapped to Initiate? Check for "Mapped order to NoOp action, skipping" or ORDER_MAPPING_FAILED
2. **Mapper conditions** (cobi-v2 executor/mapper.go): Check conditions in priority order:
   - `init_skipped_confirmations_not_met`: Source chain confirmations insufficient. Check current_confirmations vs required_confirmations. If watcher lag: check confirmation_updater.
   - `init_skipped_filled_amount_mismatch`: filledAmount != orderAmount. User sent wrong amount.
   - `init_skipped_deadline_passed_within_refund_buffer`: Deadline exceeded. 30-min buffer before instant refund triggered.
   - `init_skipped_price_threshold_breached`: Price moved >1% (priceDropThreshold). Wait for recovery or increase threshold.
   - `init_skipped_price_fetch_failed`: Fiat price API down. Transient, retries on next 2s cycle.
   - `init_blocked_by_validation`: External validation API rejected the order.
3. **BatcherWallet**: Check wallet UTXO balance. `insufficient funds: have X, need Y` → fund wallet.
4. **Fee estimation**: All fee estimators failed → check mempool.space and blockstream API connectivity.
5. **Tx submission**: Check for `bad-txns-inputs-missingorspent` (UTXO conflict, auto-recovers with RBF), `txn-mempool-conflict` (retries automatically), `ErrHighFeeEstimate` (fee rate exceeds MaxFeeRate).

### 9.2 UserRedeemPending Investigation (Bitcoin source)
1. Check btc-relayer: Did it attempt the redeem? 
2. Check relayer validation: Order must be >24 hours old. Source must have initiate_tx_hash. Destination refunded if initiated.
3. Check credentials retrieval: Can the relayer get private keys from credentials service?
4. Check fee estimation and wallet balance for the refund tx.

### 9.3 SolverRedeemPending Investigation (Bitcoin source)
1. Check executor shouldRedeem condition: SourceSwap.RedeemTxHash == "" && DestSwap.Secret != ""
2. Is the secret available from destination chain?
3. Check executor wallet balance for redeem tx
4. Check if HTLC secret validation passes: sha256(secret) == htlc.SecretHash

### 9.4 Refunded Investigation
1. Check if refund or instant refund:
   - Instant refund: requires SACP bytes. `instant_refund_skipped_no_sacp_bytes` = SACP missing, manual refund needed after timelock.
   - Timelock refund: `refund_skipped_timelock_not_expired` with blocksRemaining = need more blocks.
2. Why wasn't order completed before refund? Trace full lifecycle.

## 10. Failure → Remediation Patterns

| Failure Pattern | Log Message | Root Cause | Operator Remediation |
|---|---|---|---|
| Confirmations not met | init_skipped_confirmations_not_met | Source chain init not confirmed enough | Wait for blocks. Check watcher confirmation_updater and indexer health. |
| Filled amount mismatch | init_skipped_filled_amount_mismatch | User sent wrong amount to HTLC | Order cannot proceed. Manual intervention or user re-initiate. |
| Deadline passed | init_skipped_deadline_passed_within_refund_buffer | Order deadline exceeded + 30min buffer | If SACP bytes present: instant refund auto-triggered. If BTC source without SACP: manual refund needed. |
| Price threshold breached | init_skipped_price_threshold_breached | Price moved >1% (priceDropThreshold) | Wait for price recovery. Increase priceDropThreshold env var for stablecoin pairs. |
| Price fetch failed | init_skipped_price_fetch_failed | Fiat price API unreachable | Check fiat price service. Transient; retries on next 2s cycle. |
| Validation API rejected | init_blocked_by_validation | External validation/screener flagged the order | Check validation API service. Review the rejection reason in logs. |
| Insufficient funds | "insufficient funds: have X, need Y" | BatcherWallet doesn't have enough UTXOs | Fund the batcher wallet address. |
| Fee too high | ErrHighFeeEstimate | Required fee rate exceeds MaxFeeRate | Wait for mempool to cool down. Or increase MaxFeeRate in batcher config. |
| All fee estimators failed | "fee suggestion: error" | mempool.space and blockstream APIs unreachable | Check network connectivity to fee APIs. |
| UTXO conflict | bad-txns-inputs-missingorspent | UTXOs used were already spent (race/reorg) | Auto-recovers via RBF resubmission. |
| Mempool conflict | txn-mempool-conflict | Conflicting tx in mempool | Retries automatically on next batcher PTI cycle. |
| RBF depth exceeded | "build RBF depth exceeded" | Fee keeps increasing, needs more UTXOs | Fund the wallet. Check fee rate volatility. |
| CPFP depth exceeded | "build CPFP depth exceeded" | Similar to RBF depth issue | Fund the wallet. |
| Invalid secret on redeem | ErrInvalidSecret | sha256(secret) != htlc.SecretHash | Investigate orderbook data integrity. Possible data corruption. |
| Refund too early | "need more N blocks to refund" | currentTip < blockHeight + timelock | Wait for more blocks. |
| Invalid SACP for instant refund | ErrSACPInvalid* errors | Pre-signed SACP tx doesn't match current HTLC state | Regenerate SACP. Check if HTLC was funded after SACP generation. |
| Cache read failure | cache_read_failed | LevelDB error, action dropped | Check disk space and LevelDB health. Action retries on next cycle if cache recovers. |
| Chain worker not found | chain_worker_not_found | No executor for chainAsset key | Check config. Ensure BitcoinHTLCActionExecutor is registered for the chain+asset pair. |
| Swap config conversion failed | btc_swap_config_conversion_failed | Invalid hex in Redeemer/Initiator/SecretHash or zero Amount | Check order data integrity upstream. |
| Watcher address not in cache | No initiate event detected | pending_swaps_poller hasn't loaded swap yet | Check poller interval. ZMQ replays will re-detect after cache populated. |
| Watcher screener failure | "Failed to screen tx inputs" | Screener API error (not blacklist, API failure) | Check screener service. Deposits silently dropped until screener recovers. |
| Watcher blacklisted address | "TX has inputs from a blacklisted address" | Screener flagged input address | Order marked blacklisted. Will not be processed by executor. |
| Indexer timeout | Context deadline exceeded | Electrs/esplora slow or unreachable (5s timeout) | Check indexer service load and connectivity. |
| Relayer: order too recent | "Order is newer than 24 hours" | Safety check for premature refunds | Wait. Relayer deliberately waits 24 hours. |
| Relayer: dest not refunded | "Destination swap has not been refunded" | COBI hasn't refunded destination yet | Wait for COBI to refund destination. Check executor refund logic. |
| Relayer: timelock not expired | "Source swap timelock not expired yet" | block_height + timelock > current_tip | Wait for more blocks. |
