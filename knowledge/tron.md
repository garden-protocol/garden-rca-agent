## 1. Service Architecture Overview

### What Each Service Does

**tron-watcher** (Rust, async/tokio)
Monitors HTLC contract events on Tron via JSON-RPC (`eth_getLogs`, `eth_blockNumber`). Reads pending swaps from the shared PostgreSQL database (`swaps` table), fetches their on-chain state via `eth_call` (batch RPC for `getOrder` selector `0x9c3f1e90`), computes block ranges where status mismatches exist between DB and chain, fetches logs for those ranges, parses `GardenHTLCv3` events (Initiated, InitiatedWithDestinationData, Redeemed, Refunded), validates them, writes state back to DB, and tracks confirmation counts. Uses a `moka` in-memory cache for confirmation tracking (1-hour TTL). Requires **2 block confirmations** before considering a swap finalized.

**tron-executor** (TypeScript/Bun, Effect-TS)
The solver-side on-chain action engine. Polls pending orders from the solver engine's orders API, maps them to HTLC actions (Initiate, InitiateWithSignature, Redeem, Refund, InstantRefund, NoOp) via `OrderMapper`, simulates transactions via `TxSimulatorService`, then submits batches via `TronActionExecutor` (multicall). Handles TRC20 allowance checks and approval. Registers with the executor registry on startup and runs a validator server (port 4525).

**tron-relayer** (TypeScript/Bun, Effect-TS, Hono HTTP framework)
User-facing relay service. Two concurrent subsystems:
- **HTTP Server** (Hono): Handles `GET /:version/:orderId?action=Initiate` for calldata generation and `PATCH /:version/:orderId?action=Initiate|Redeem` for execution. Uses JWT auth via `authMiddleware`. Supports initiate-with-signature (user signs, relayer submits) and redeem (user provides secret).
- **Redeemer Service**: Background poller that fetches pending orders, filters for redeemable ones (destination is Tron, source initiated, not already redeemed/refunded), retrieves secrets from a credentials provider, builds `TronActionRequest`s, simulates, and submits in batches via `TronActionExecutor`.

Also runs a `TransactionExecutorService` which queues requests from HTTP handlers and processes them in batches with simulation.

### Order Lifecycle on Tron

```
[User]  ---(EIP-712 signature)---> [tron-relayer PATCH /:v/:orderId?action=Initiate]
                                          |
                              { body: { signature } }
                                          |
                         _getInitiateWithSignatureProgram()
                            - fetches MatchedOrder from orderbook
                            - builds TronActionRequest (InitiateWithUserSignature)
                            - submits to TransactionExecutorService queue
                                          |
                              TransactionExecutor.processBatch()
                                          |
                              TxSimulator.simulate() -> valid/invalid split
                                          |
                              TronActionExecutor.submit(validRequests)
                                          |
                              Multicall on-chain TX -> txHash
                                          |
                          [tron-watcher picks up Initiated event]
                            - validates event (redeemer, timelock, asset, chain)
                            - updates swaps.filled_amount, initiate_tx_hash, initiate_block_number
                            - tracks confirmations until >= 2
                                          |
                          [tron-executor picks up pending order]
                            - OrderMapper decides action (Initiate dest / Redeem source / Refund / NoOp)
                            - simulates + submits via multicall
                                          |
                          [tron-relayer redeemer picks up redeemable order]
                            - retrieves secret from credentials provider
                            - builds Redeem TronActionRequest
                            - simulates + submits via multicall
                                          |
                          [tron-watcher picks up Redeemed event]
                            - writes secret, redeem_tx_hash, redeem_block_number
```

| `src/tron/primitives.rs` | `TronAddress` — Base58 <-> hex conversion, checksum validation |

### Watcher Cycle (`iter_next`)

1. **Fetch DB swaps**: Query `swaps` table for chain's pending swaps (uninitiated within deadline buffer, OR initiated but not redeemed/refunded) + confirmation tracking cache
2. **Fetch on-chain swaps**: Batch `eth_call` to `getOrder(orderID)` on HTLC contracts, decode 192-byte ABI response into `OnChainSwap` (initiator, redeemer, initiated_at, timelock, amount, fulfilled_at)
3. **Sync check**: Verify current block >= max block in on-chain data (prevents stale node issues)
4. **Calculate block ranges**: Compare on-chain status vs DB status; collect mismatched block numbers, split into ranges by `max_block_span`
5. **Fetch events**: `eth_getLogs` for HTLC contract addresses over calculated ranges (up to 6 concurrent workers, 4 ranges per worker minimum)
6. **Parse events**: Decode `GardenHTLCv3Events` — `Initiated`, `InitiatedWithDestinationData`, `Redeemed` (extracts secret), `Refunded`
7. **Validate events**: Match swap_id to known swaps, verify redeemer/timelock/asset/chain
8. **Store events**: Write to DB in transaction — `handle_inits` (UPDATE filled_amount, initiate_tx_hash, block, timestamp WHERE swap_id + chain + asset + redeemer + timelock), `handle_redeems` (UPDATE redeem_tx_hash, block, timestamp, secret), `handle_refunds` (UPDATE refund_tx_hash, block, timestamp)
9. **Update confirmations**: Track block confirmations for fulfilled swaps (needs >= 2 confirmations); track initiated swap confirmations and write `current_confirmations = LEAST(required_confirmations, calculated)` to DB

### DB Queries

**get_swaps** (what the watcher watches):
```sql
SELECT s.* FROM swaps s
WHERE s.chain = $1
  AND (
    (s.initiate_tx_hash = '' AND s.created_at + ($2 * INTERVAL '1 second') > NOW())
    OR
    (s.initiate_tx_hash != '' AND s.redeem_tx_hash = '' AND s.refund_tx_hash = '')
  )
```

**handle_inits** UPDATE:
```sql
UPDATE swaps SET filled_amount=$1, initiate_tx_hash=$2, initiate_block_number=$3, initiate_timestamp=$4
WHERE swap_id=$5 AND chain=$6 AND asset=$7 AND redeemer=$8 AND timelock=$9
```

**handle_redeems** UPDATE:
```sql
UPDATE swaps SET redeem_tx_hash=$1, redeem_block_number=$2, redeem_timestamp=$3, secret=$4
WHERE swap_id=$5 AND chain=$6 AND asset=$7
```

**handle_refunds** UPDATE:
```sql
UPDATE swaps SET refund_tx_hash=$1, refund_block_number=$2, refund_timestamp=$3
WHERE swap_id=$4 AND chain=$5 AND asset=$6
```

### Key Files & Functions

| File | Purpose |
|------|---------|
| `main.ts` | Entry point: resolves config, creates TronWeb, initializes FiatProvider, PendingOrdersProvider, OrderMapper, TronActionExecutor, ValidatorService, Executor; starts executor loop |
| `executor/executor.ts` | `Executor` class — `run()` polls on Schedule.spaced(5s); `executeLoop()`: get pending orders -> get block number if needed -> map to action requests -> filter cached -> check allowance/approve -> simulate -> batch -> execute |
| `orders/orderMapper.ts` | `OrderMapper.map()` — determines action via priority: canRefund > canInitiate > canRedeem > canInstantRefund > NoOp. Checks source confirmations, destination state, blacklist, price threshold, deadline |
| `orders/types.ts` | `MatchedOrder`, `Swap` types |
| `validator/server.ts` | Validator HTTP server (port 4525) for executor registry health checks |
| `validator/register.ts` | Registers executor with the executor registry |
| `quote/fiatProvider.ts` | Fetches token prices for price-drop threshold checks |

### Executor Polling Cycle (`executeLoop`)

1. **Get pending orders**: Fetch from pendingOrdersUrl (orders API)
2. **Get block number**: Only if there are refundable swaps (destination initiated but not completed)
3. **Map orders to actions**: For each order, `OrderMapper.map()` determines the action:
   - **Refund**: Destination initiated, timelock expired (`blockNum >= initiate_block + timelock`)
   - **Initiate** (InitiateWithSignature): Source initiated + confirmed + filled, destination not initiated, not blacklisted, not instant-refundable, price within threshold
   - **Redeem**: Source not completed, destination has secret + redeem_tx_hash
   - **InstantRefund**: Source initiated but should instant refund (partial fill, price drop, deadline expired, or destination refunded without source refund)
   - **NoOp**: Default fallback
4. **Filter cached**: Skip orders already in the 1-hour execution cache
5. **Check allowance**: For each action, check TRC20 allowance via `trc20Contract.allowance(owner, spender).call()`, approve `MAX_UINT256` if insufficient
6. **Simulate**: `TxSimulatorService.simulate(requests)` -> valid/invalid split
7. **Cache invalid**: Store in cache to avoid retrying
8. **Batch & Execute**: Split valid requests into batches of 10, submit each batch via `TronActionExecutor.submit(batch)` -> txHash
9. **Cache executed**: Store in cache with txHash

### Action Type Mapping (OrderMapper)

| Condition | Action | Target Swap |
|-----------|--------|-------------|
| Dest initiated + timelock expired | `Refund` | destination_swap |
| Source initiated + confirmed + filled, dest not initiated, not blacklisted | `InitiateWithSignature` | destination_swap |
| Source not completed, dest has redeem + secret | `Redeem` | source_swap |
| Source initiated, should instant refund (price drop/deadline/partial fill) | `InstantRefund` | source_swap |
| None of the above | `NoOp` | none |

### HTTP API

**GET /:version/:orderId?action=Initiate**
- Returns initiate calldata for user to sign
- Response: `{ status: "Ok", result: { initiate_transaction, approval_transaction?, typed_data? } }`

**PATCH /:version/:orderId?action=Initiate**
- Body: `{ signature: "0x..." }`
- Submits InitiateWithUserSignature via TransactionExecutor queue
- 30-second timeout on deferred response
- Response: `{ status: "Ok", result: "txHash" }`

**PATCH /:version/:orderId?action=Redeem**
- Body: `{ secret: "0x..." }`
- Validates secret matches order's secret_hash
- Validates timelock not expired: `currentBlock < initiate_block + timelock`
- Submits Redeem via TransactionExecutor queue
- Response: `{ status: "Ok", result: "txHash" }`

### Redeemer Service Flow

1. **Poll**: Fetch pending orders from `pendingOrdersProviderUrl` every 5s
2. **Filter**: Via `filterOrders()`:
   - Skip if in processedOrders cache (already handled)
   - Skip if in nonProcessableOrders cache (no credentials)
   - Validate: destination is Tron (chain = `tron_testnet` or `tron`), source initiated + confirmed, source not refunded, destination initiated, not already redeemed/refunded
3. **Get credentials**: Extract secret_hashes, call credentials provider, partition into with/without credentials, cache non-processable ones
4. **Build requests**: For each order with credentials, build TronActionRequest with `TronHTLCActionType.Redeem` + `addHexPrefix(secret)`
5. **Simulate**: Via `TxSimulatorService`, cache invalid request IDs
6. **Submit**: Batch (size 20) submit via `TronActionExecutor.submit(batch)`, cache successful IDs

## 5. HTLC Contract Interaction

### Contract Type
Uses `GardenHTLCv3` — the same Solidity ABI as EVM chains but deployed on Tron. Tron is EVM-compatible at the smart contract level (uses `eth_` JSON-RPC methods). Contracts handle **TRC20 tokens** (not native TRX) through the HTLC pattern:

- The HTLC contract has a `token()` function returning the TRC20 token address
- Approval is needed: `trc20.approve(htlcAddress, MAX_UINT256)`
- Native token address is the zero address: `T9yD14Nj9j7xAB4dbGeiX9h8unkKHxuWwb`

### Key Contract Functions

| Function | Selector | Purpose |
|----------|----------|---------|
| `getOrder(bytes32)` | `0x9c3f1e90` | Read on-chain swap state (initiator, redeemer, initiated_at, timelock, amount, fulfilled_at) |
| `initiate(...)` | via ABI | Initiate a new HTLC swap |
| `initiateOnBehalf(...)` | via ABI | Initiate with delegator pattern |
| `redeem(bytes32, bytes32)` | via ABI | Redeem with secret |
| `refund(bytes32)` | via ABI | Refund after timelock |
| `token()` | via ABI | Get the TRC20 token address |

### On-Chain Swap Structure (ABI-encoded, 192 bytes = 6 * 32)

| Offset | Field | Type |
|--------|-------|------|
| 0-31 | initiator | address (bytes12 padding + 20 bytes) |
| 32-63 | redeemer | address (bytes12 padding + 20 bytes) |
| 64-95 | initiated_at | uint256 (block number) |
| 96-127 | timelock | uint256 (blocks) |
| 128-159 | amount | uint256 |
| 160-191 | fulfilled_at | uint256 (block number, 0 if not fulfilled) |

### Events Watched

| Event | Fields | DB Action |
|-------|--------|-----------|
| `Initiated(orderID)` | orderID, block, tx_hash | SET filled_amount, initiate_tx_hash, initiate_block_number, initiate_timestamp |
| `InitiatedWithDestinationData(orderID)` | Same as Initiated | Same as Initiated |
| `Redeemed(orderID, secret)` | orderID, secret, block, tx_hash | SET redeem_tx_hash, redeem_block_number, redeem_timestamp, secret |
| `Refunded(orderID)` | orderID, block, tx_hash | SET refund_tx_hash, refund_block_number, refund_timestamp |

### Address Format
Tron uses Base58Check addresses (prefix `T`). The watcher converts between:
- **Base58** (e.g., `TCx2ghnhkdKB8L3mj4qSyjB4VETJDWe42q`) — used in DB and config
- **Hex** (e.g., `0x20af5981eb329afec812d9fa0a877ca909ebb64e`) — used in JSON-RPC

Conversion: Base58 decode -> strip version byte (0x41) and 4-byte checksum -> hex encode. Reverse: hex decode -> prepend 0x41 -> double SHA256 checksum -> Base58 encode.

### Multicall
Both executor and relayer submit batches via `TronActionExecutor` which uses a multicall contract (`multicallAddress` in config). This batches multiple HTLC actions into a single transaction.

## 6. Error Patterns & Log Messages

### Watcher Errors (Rust/tracing)

| Error Message | Cause | Severity |
|---------------|-------|----------|
| `"Watcher encountered an error in chain '{chain}': {error}"` | Any error in `iter_next()` cycle | ERROR — logged + continues |
| `"Failed to store swap events: {err}"` | DB write failure during event processing | ERROR — skips batch, continues |
| `"Failed to update confirmation: {err}"` | Confirmation update DB failure | ERROR — skips, continues |
| `"Error processing event stream: {e}"` | Event fetch/parse failure | ERROR — skips batch, continues |
| `"Failed to parse swap event: {err}"` | Individual event decode failure | ERROR — skips event, continues |
| `"No affected rows for initiate/redeem/refund event, swap {id}"` | DB UPDATE matched no rows (swap_id + chain + asset + redeemer mismatch) | ERROR — **missed_init candidate** |
| `"Invalid initiation/redeem/refund event for swap '{id}'"` | Event validation failed (redeemer/timelock/asset/chain mismatch) | WARN |
| `"Failed to convert swap '{id}' to OnchainRequest: {error}"` | Invalid contract address in swap | WARN |
| `"Invalid or empty onchain swap found for swap id: '{id}'"` | `getOrder` returned all zeros | INFO |
| `"RPC error: {error}"` | JSON-RPC returned error in batch response | ERROR |
| `"Received empty result ('0x') from eth_call"` | No on-chain order exists for swap_id | ERROR |
| `"Failed to send request: {e}"` | HTTP connection failure to Tron RPC | ERROR |
| `"Failed to deserialize contract logs: {e}"` | Log JSON parse failure | ERROR |

### Executor Errors (TypeScript/winston)

| Error Message | Cause | Severity |
|---------------|-------|----------|
| `"tron-executor: Failed to execute requests"` | Any error in `executeLoop()` | ERROR |
| `"tron-executor: Failed to execute batch"` | Batch submission failure | ERROR |
| `"tron-executor: Failed to map order to action request"` | OrderMapper error (invalid swap data) | ERROR |
| `"tron-executor: Validator rejected Initiate action, skipping"` | Validator service rejected the initiate | WARN |
| `"Failed to approve: {error}"` | TRC20 approve transaction failed | ERROR |
| `"Failed to check current allowance for {owner} on {spender}"` | Allowance RPC call failed (retries 4x at 15s intervals) | ERROR |
| `"Failed to get trc20 contract address: {error}"` | `htlcContract.token().call()` failed | ERROR |

### Relayer Errors (TypeScript/Effect)

| Error Message | Cause | Severity |
|---------------|-------|----------|
| `"Failed to start TRON Relayer: {error}"` | Startup failure | FATAL |
| `"Failed to execute requests"` | Redeemer or TransactionExecutor loop error | ERROR |
| `"Batch submission failed"` | Redeemer batch submit failure | ERROR |
| `"Batch processing failed: {error}"` | TransactionExecutor batch failure | ERROR |
| `"Invalid Execution Request for request {id}"` | Simulation rejected the request | ERROR (returned to caller) |
| `"Order not found"` | Orderbook lookup returned nothing | 404 |
| `"Cannot redeem, order's timelock has expired"` | `currentBlock >= initiateBlock + timelock` | 400 |
| `"Missing or invalid parameter: {name}"` | Missing `signature` or `secret` in request body | 400 |
| `"Order already executed"` | Cache hit — action already submitted for this order | 400 |

## 7. Investigation Playbook

### Alert: missed_init (Source initiated on-chain, but watcher didn't write to DB)

**Symptoms**: Order shows source initiate on-chain (TronScan) but DB has empty `initiate_tx_hash`.

**Investigation steps**:
1. **Check watcher logs** for the swap_id:
   - Search for `"No affected rows for initiate event, swap {swap_id}"` — means the UPDATE's WHERE clause didn't match. Check redeemer, timelock, asset, chain values in DB vs on-chain event.
   - Search for `"Invalid initiation event for swap '{swap_id}'"` — validation failed. Log shows expected vs actual redeemer/timelock/asset/chain.
   - Search for `"Failed to convert swap '{swap_id}' to OnchainRequest"` — invalid TronAddress in DB's `asset` field.
   - Search for `"Invalid or empty onchain swap found for swap id"` — `getOrder` returned zeros; HTLC contract may not have the order yet or wrong contract address.
2. **Check if watcher is watching the right contract**: Compare `contract_addresses` in Settings.toml with the actual HTLC contract used by the order (DB `asset` field is the HTLC contract address on Tron).
3. **Check block range coverage**: The watcher only queries blocks where on-chain state differs from DB state. If `getOrder` shows `initiated_at=0` (order not visible on-chain yet), the watcher won't compute block ranges for it.
4. **Check RPC health**: Search for `"Failed to send request"` or `"RPC error"`. Tron RPC can be flaky — the watcher retries 3 times with backoff.
5. **Check DB swap state**: The `get_swaps` query only returns swaps within `deadline_buffer_secs` of creation (default 120s). If the swap is too old and was never initiated, the watcher stops watching it.

### Alert: stuck_order (Order stuck in initiated state, not progressing)

**Symptoms**: Source initiated + confirmed, but destination not initiated.

**Investigation steps**:
1. **Check executor logs** for the order's create_id:
   - `"tron-executor: No pending orders to process"` — orders API returned nothing for this order. Check the pendingOrdersUrl.
   - `"tron-executor: Validator rejected Initiate action"` — validator service rejected it. Check validator service health.
   - `"Source not ready"` / `"Order is blacklisted"` / `"Order is not screened yet"` — OrderMapper determined the order isn't ready.
2. **Check confirmation status**: Source swap needs `current_confirmations >= required_confirmations` AND `filled_amount == amount`. Query DB:
   ```sql
   SELECT swap_id, current_confirmations, required_confirmations, filled_amount, amount
   FROM swaps WHERE swap_id = '{source_swap_id}';
   ```
3. **Check if executor is caching it**: The executor caches processed orders for 1 hour. If the order was previously attempted and failed simulation, it's cached as invalid. The executor must restart or wait for cache expiry.
4. **Check relayer redeemer** if destination Tron: The redeemer only processes orders where `destination_swap.initiate_tx_hash` exists. If destination init hasn't happened yet, check executor (step 1).
5. **Check for allowance issues**: Search executor logs for `"Failed to approve"` or `"Failed to check current allowance"`. The executor needs TRC20 approval before initiating.

### Alert: stuck_order (Destination initiated, but not redeemed)

**Investigation steps**:
1. **Check relayer redeemer logs**:
   - `"Order {id} is an already processed order, skipping"` — already in cache.
   - `"Order {id} is non processable, skipping"` — credentials (secret) not available from credentials provider.
   - `"destination swap is not a tron swap"` — chain mismatch.
   - `"source swap has no initiate tx hash"` / `"source swap has unconfirmed initiate tx hash"` — source not yet confirmed.
   - `"order has no initiate tx hash"` — destination not initiated in DB.
   - `"order has already been redeemed/refunded"` — already done.
2. **Check credentials provider**: The redeemer needs the secret from the credentials provider URL. If it returns nothing for the secret_hash, the order gets cached as non-processable.
3. **Check executor logs for redeem**: If source chain is Tron, the executor handles redeem. Look for `"Cannot redeem"` in OrderMapper logic — needs `dest.redeem_tx_hash` + `dest.secret` set in DB (dest watcher must have seen the redeem event first).

### Alert: refunded (Order refunded unexpectedly)

**Investigation steps**:
1. **Check watcher logs** for the refund event — search for `"Updated swap {id} with refund event"`.
2. **Check timelock**: Was the timelock expired? Compare `initiate_block_number + timelock` vs the refund block.
3. **Check executor logs** for refund action:
   - The executor refunds when `blockNum >= initiate_block + timelock` for destination swaps.
   - Check if `canRefund` was triggered — log `"Cannot refund"` should not appear.
4. **Check for instant refund**: Executor triggers `InstantRefund` on source when:
   - Partial fill (`amount != filled_amount`)
   - Price dropped beyond threshold (default 1%)
   - Deadline expired (order `deadline` or fallback 1 hour)
   - Destination refunded but source not

### General Debugging Tips

1. **Tron address format**: Always check if address comparison is correct. The watcher uses `strings_match()` with `to_lowercase()` — Base58 is case-sensitive, but the watcher normalizes to lowercase.
2. **Block times**: Tron produces blocks every ~3 seconds. Timelock is in blocks, not seconds.
3. **Multicall failures**: If a batch submission fails, all actions in the batch fail. Check if one bad action is poisoning the batch.
4. **Cache TTLs**: Watcher confirmation cache = 1 hour. Executor execution cache = 1 hour. Relayer processed orders cache = 30 minutes. Relayer non-processable orders cache = 30 minutes (from `OrderCacheService`).
5. **RPC endpoint**: Tron uses `https://api.trongrid.io/jsonrpc` (mainnet) or `https://api.shasta.trongrid.io/jsonrpc` (testnet). The `/jsonrpc` suffix is critical — it's the EVM-compatible JSON-RPC endpoint.
6. **Confirmation requirement**: The watcher requires **2 block confirmations** (`MINIMUM_CONFIRMATIONS_REQUIRED = 2`) before removing fulfilled swaps from the confirmation tracking cache.
