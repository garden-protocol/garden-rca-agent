# Alpen Chain Knowledge Base

Generated by Claude Code from source. Covers 2 repos:
- **watcher** — `alpen-watcher` (Go, branch: feat/alpen)
- **executor** — `alpen-executor` (Go, branch: feat/alpen)

Shared library: `blockchain` (Go, branch: feat/alpen) — HTLC scripts, Taproot wallet, indexer client, batcher

**Chain Nature**: Alpen is a **Bitcoin Layer 2 chain (UTXO-based)**. It uses the same Taproot/Tapscript HTLC structure as Bitcoin. Identified by `NewUtxoChain()` with `TypeUTXOBased = "utxo"` in the watcher, and `ContractAddress = "primary"` keyword (not an actual hex address) in the executor config.

## 1. Service Architecture Overview

### What Each Service Does

**alpen-watcher** (Go, GORM + btcd)
Monitors the Alpen blockchain for HTLC events (Initiated, Redeemed, Refunded) and updates swap state in a shared PostgreSQL database. Uses an Electrs-compatible indexer API to fetch transaction data. Parses Taproot witness scripts to classify events. Runs N worker goroutines with a job scheduler. Tracks confirmations every 5 seconds.

**alpen-executor** (Go, btcd + go-ethereum hybrid)
Hybrid executor that handles both UTXO-based (Alpen/Bitcoin) and EVM-based chains. Polls the orderbook API for pending orders every 2 seconds, maps them to HTLC actions, and executes on-chain. Uses `BitcoinHTLCActionExecutor` for Alpen (UTXO path) and `EVMHTLCActionExecutor` for EVM chains. Batched transaction support with RBF strategy.

### Dependencies
- `github.com/btcsuite/btcd` — Bitcoin protocol, Taproot scripts, address encoding
- `gorm.io/gorm` — PostgreSQL ORM (watcher)
- `github.com/catalogfi/blockchain/btc` — HTLC wallet, batcher, indexer client
- `github.com/ethereum/go-ethereum` — EVM chain interaction (hybrid executor)
- `github.com/syndtr/goleveldb/leveldb` — Action caching (executor)

### Order Lifecycle on Alpen

```
Watcher polls Electrs indexer for HTLC address transactions →
Classifies: VOUTs to address = Initiated, VINs from address = Redeemed/Refunded →
Parses Taproot witness (RedeemLeaf/RefundLeaf/MultiSigLeaf) → Updates PostgreSQL →
Executor polls orderbook API → Maps orders to actions → Batch execution via BTCWallet →
Transaction broadcast via Electrs API
```

## 2. Watcher Flow

### Key Files
- `watcher/watcher.go` — main watcher loop, event processing, state machine
- `watcher/error.go` — error type classification (NonRecoverable/Recoverable/Ignorable)
- `blockchain/btc/event.go` — HTLC event detection from indexer transactions
- `blockchain/btc/htlc.go` — HTLC wallet operations
- `blockchain/btc/scripts.go` — Taproot script construction and leaf validation
- `blockchain/btc/indexer.go` — Electrs indexer client (HTTP)
- `blockchain/blockchain.go` — chain name constants, UTXO chain type
- `store/store.go` — database operations (GORM)
- `model/model.go` — data models, chain constants
- `config/config.go` — watcher configuration

### Startup
1. `NewWatcher()` initializes with Store (PostgreSQL), HTLCClient (wrapping IndexerClient), event/blacklist/job caches, chain config from mode (localnet/testnet/mainnet)
2. `Start()` launches worker goroutines (`startWorkers()`), job scheduler (`scheduleJobs()` on ticker), confirmation tracker (`handleConfirmations()` every 5s)

### Job Scheduling
On each tick, `fetchAndQueueJobs()` queries `store.Observables()` for swaps where:
- `chain = alpen_* AND asset = primary AND amount > 0`
- Created within last 7 days
- Either: no initiate_tx_hash (within order deadline), OR has initiate but no redeem/refund block numbers

Each observable is queued as a `Job` if not in job cache.

### Event Processing (`processObservable()`)
For each observable HTLC address:
1. Check blacklist cache → skip if blacklisted
2. Decode address using `btcutil.DecodeAddress` with chain params
3. Fetch HTLC events via `htlcClient.HTLCEvents()` (Electrs indexer)
4. Remove duplicate events (known bug workaround for partial inits)
5. Check event cache → skip if unchanged
6. Filter into: Initiated, Redeemed, Refunded
7. Process each:
   - `handleInits()` — accumulate filled amount, track block numbers, set confirmations
   - `handleRedeems()` — extract secret, set redeem tx hash and block
   - `handleRefundOrRedeem()` — distinguish refunds from redeems-disguised-as-refunds
8. Handle timestamps — set detected timestamps in additional_data
9. Blacklist check — if init timestamp before created_at, blacklist the order
10. Update swap in DB via `store.UpdateSwap()`

### Event Detection from Chain (`blockchain/btc/event.go`)
`HTLCEvents()` function:
- Fetches all transactions for HTLC address via Electrs indexer
- **VOUTs going TO the HTLC address** = `HTLCInitiated` event (amount, block height)
- **VINs spending FROM the HTLC address** = Redeemed or Refunded, determined by parsing witness script:
  - `IsRedeemLeaf()` — OP_SHA256 + secret_hash + OP_EQUALVERIFY + redeemer_pubkey + OP_CHECKSIG
  - `IsRefundLeaf()` — timelock + OP_CHECKSEQUENCEVERIFY + OP_DROP + initiator_pubkey + OP_CHECKSIG
  - `IsMultiSigLeaf()` — 2-of-2 multisig (instant refund path)

### Confirmation Tracking
Every 5 seconds:
1. Get latest block height from indexer
2. Update `current_confirmations` for all swaps where `required_confirmations > current_confirmations` and `initiate_block_number != 0`

### Chain Params
- `Alpen` → `MainNetParams`
- `AlpenSignet` → `TestNet3Params`
- `AlpenRegtest` → `RegressionNetParams`

### Configuration
```json
{
  "mode": {"mode": "mainnet|testnet|localnet"},
  "database": {"db_url": "postgres://..."},
  "watcher": {"interval": 60, "workers": 4},
  "alpen": {"rpc": "http://electrs-url:30000", "chain": "alpen", "asset": "primary"},
  "integrator_info": {"api_url": "", "allowed_mismatch_percent": 3},
  "order_deadline": 3600
}
```

## 3. Executor Flow

### Key Files
- `cmd/main.go` — entry point, wallet init, HTTP server
- `executor/executor.go` — main execution loop (2s ticker)
- `executor/mapper.go` — order-to-action mapping logic
- `executor/validator.go` — external order validation API client
- `bitcoin/actions.go` — Bitcoin/Alpen HTLC action executor
- `server/server.go` — HTTP validation server
- `actions/actions.go` — HTLC action type definitions
- `ethereum/htlc.go` — EVM HTLC swapper (for hybrid chains)
- `orderbook/models.go` — order/swap data models

### Startup
1. Load config from JSON, await keystore password on port **4428**
2. Initialize Bitcoin wallet with BatcherWallet (RBF strategy)
3. Initialize EVM HTLC swappers for non-Bitcoin chains
4. Create executor with all action executors
5. Start HTTP server on port **4527** (validation + accounts)
6. Register with order validator API
7. Call `executor.Start()`

### Main Execution Loop (`executor.go`)
Two tickers:
- **2-second**: `executeOrders(ctx)` — main loop
- **1-hour**: cache cleanup (24h TTL)

Pipeline:
1. Fetch pending orders from orderbook API
2. Filter whitelisted initiators
3. Get latest block numbers for refund calculations
4. Remove duplicates + blacklisted
5. Map orders to actions via `OrderToActionMapper`
6. Filter cached actions (skip already-executed)
7. Split by chain, execute concurrently

### Order-to-Action Mapping (`mapper.go`)

| Condition | Action |
|-----------|--------|
| `destination.initiate_tx_hash == ""` | **Init** (if confirmations ok, amount ok, deadline ok, price ok) |
| `source.redeem_tx_hash == ""` AND `destination.secret != ""` | **Redeem** source swap |
| `destination.refund_tx_hash == ""` AND `destination.initiate_tx_hash != ""` AND timelock expired | **Refund** dest swap |
| `destination.refund_block > 0` AND no source redeem/refund | **Instant Refund** source (SACP) |
| deadline passed + 30min buffer AND source is Alpen | **Instant Refund** instead of Init |
| Otherwise | **NoOp** |

Init gating:
1. Source confirmations >= required
2. Filled amount == expected
3. Deadline not passed (default 12 hours)
4. Price within threshold (default 1%)
5. External validator API approval (script address verification)

### Alpen Chain Detection
`isAlpenChain()` checks `strings.Contains(chainAsset, orderbook.ChainAlpen)`. Alpen uses `BitcoinHTLCActionExecutor` (UTXO path), NOT the EVM path. The keyword `ContractAddress = "primary"` signals a UTXO chain.

### Action Execution (`bitcoin/actions.go`)
1. Filter already-initiated (on-chain check via `swapper.IsInitiated()`)
2. Batch execute multiple HTLC actions in one tx
3. Fall back to individual if batch fails
4. Maps to: `InitiateHTLCAction`, `RedeemHTLCAction`, `RefundHTLCAction`, `InstantRefundHTLCAction`

## 4. HTLC Interaction

### HTLC Type: Taproot Script V2 (same as Bitcoin)

Three leaf scripts in a Taproot script tree:

**Leaf 1 — Redeem:**
```
OP_SHA256 <secret_hash> OP_EQUALVERIFY <redeemer_xonly_pubkey> OP_CHECKSIG
```

**Leaf 2 — Refund:**
```
<timelock> OP_CHECKSEQUENCEVERIFY OP_DROP <initiator_xonly_pubkey> OP_CHECKSIG
```

**Leaf 3 — Instant Refund (2-of-2 MultiSig):**
```
<initiator_pubkey> OP_CHECKSIG <redeemer_pubkey> OP_CHECKSIGADD OP_2 OP_NUMEQUAL
```

### Internal Key (NUMS Point)
BIP-341 H value + `sha256("GardenHTLC")`. Key-path spend unusable — funds only via script leaves.

### Address Type
Taproot (`btcutil.NewAddressTaproot`). The HTLC "address" IS the swap ID.

### Contract/Asset Identifier
- `ContractAddress = "primary"` → keyword signaling UTXO chain (not an actual address)
- The HTLC address is derived from script tree parameters (initiator, redeemer, secret_hash, timelock)

### Instant Refund (SACP)
Same as Bitcoin/Litecoin: user pre-signs SACP tx with `SigHashSingle|AnyoneCanPay`, executor counter-signs.

### Key Functions
| Function | File | Purpose |
|----------|------|---------|
| `Address()` | `blockchain/btc/htlc.go` | Compute Taproot address from HTLC params |
| `Initiate()` | `blockchain/btc/htlc.go` | Send BTC to HTLC Taproot address |
| `Redeem()` | `blockchain/btc/htlc.go` | Spend from redeem leaf with secret |
| `Refund()` | `blockchain/btc/htlc.go` | Spend from refund leaf after timelock |
| `Execute()` | `blockchain/btc/htlc.go` | Batch-execute multiple HTLC actions |
| `GenerateInstantRefundSACP()` | `blockchain/btc/htlc.go` | Create SACP tx for 2-of-2 refund |
| `HtlcScriptV2()` | `blockchain/btc/scripts.go` | Generate expected Taproot address for validation |

## 5. Error Patterns

### Watcher Errors
| Error Message | Source | Meaning |
|---|---|---|
| `"failed to get HTLC events for {addr}"` | `watcher.go` | Indexer API call failed |
| `"failed to decode address {addr}"` | `watcher.go` | Invalid HTLC address format |
| `"Error processing observable"` | `watcher.go` | General processing failure |
| `"Failed to update current confirmations"` | `watcher.go` | Block height fetch or DB update failed |
| `"failed to get latest block"` | `watcher.go` | Indexer unreachable |
| `"init tx timestamp is before created_at, skipping"` | `watcher.go` | Suspicious order — blacklisted |
| `"failed to update swap in database"` | `watcher.go` | DB write failure |
| `"order not found for swap id"` | `watcher.go` | Matched order lookup failed |
| `"Failed to fetch and queue jobs"` | `watcher.go` | DB observables query failed |

### Executor Errors
| Error Message | Source | Meaning |
|---|---|---|
| `"could not convert the swap into bitcoin compatible swap"` | `bitcoin/actions.go` | Malformed swap data |
| `"error checking if HTLC is initiated with secret hash"` | `bitcoin/actions.go` | Indexer error |
| `"No chain worker found for chainAsset"` | `executor/executor.go` | Config missing for chain |
| `"failed to get pending orders"` | `executor/executor.go` | Orderbook API unreachable |
| `"need more %d blocks to refund"` | `blockchain/btc/htlc.go` | Timelock not expired |
| `"invalid secret"` | `blockchain/btc/htlc.go` | Secret hash mismatch |
| `"insufficient balance"` | `blockchain/btc/indexer.go` | Wallet lacks funds |
| `"script address mismatch"` | `server/server.go` | HTLC address verification failed |
| `"source swap validation failed"` | `executor/mapper.go` | Validator rejected |
| `"nonce too low"` | `executor/executor.go` | EVM-side: already processed |
| `"order fulfilled"` | `executor/executor.go` | Already completed |

### Watcher Error Types
- `NonRecoverableError` — Fatal, should not retry
- `RecoverableError` — Transient, will retry
- `IgnorableError` — Safely skipped

## 6. Investigation Playbook

### missed_init (Source initiated, destination not)

1. **Check executor logs** for `createID` or `secretHash`:
   - `"Mapper: NoOpHTLCAction"` → decided not to act
   - `"source swap validation failed"` → validator rejected
   - `"Failed to execute batch actions"` → execution failure
2. **Confirmation status**: `current_confirmations >= required_confirmations`?
3. **Deadline check**: Order deadline + `ignoreOrderDeadline` hours passed?
4. **Price threshold**: Price volatile beyond `priceDropThreshold`?
5. **Filled amount**: `filled_amount == amount`? Partial fill won't trigger.
6. **Watcher logs**: Processing this swap? `"New HTLC Initiated event"` for address?
7. **Indexer health**: `"failed to get HTLC events"` errors?
8. **Chain config**: Alpen chain asset in `chainConfig`?
9. **Wallet balance**: Sufficient BTC? Check validation server at `:4527/validate`.
10. **Script address**: HTLC address matches expected?

### stuck_order (Not progressing)

1. **DB state**: Which swap fields populated?
   - No initiate on either → watcher not running/processing
   - Source inited, dest not → missed_init
   - Both initiated, no redeem → secret not extracted yet
   - Dest redeemed, source not → executor should redeem with secret
2. **Watcher observables**: Looks back 7 days, filtered by `order_deadline`. Aged out?
3. **Event cache**: Stale? Restart watcher to clear.
4. **Blacklist**: Init timestamp before created_at → blacklisted.
5. **Amount mismatch**: `HandleMismatchedAmount()` logic, check integrator's `allowed_mismatch_percent`.
6. **Executor cache**: LevelDB `cache.HasKey(secretHash + "_Init")` → won't retry.

### refunded

1. **Which side**: `destination_swap.refund_tx_hash` or `source_swap.refund_tx_hash`?
2. **Timelock**: `initiate_block_number + timelock <= current_block`?
3. **Instant refund**: Check for `IsMultiSigLeaf` in watcher, `"InstantRefund"` in executor.
4. **SACP validation**: `additional_data.instant_refund_tx_bytes` exists? Valid?
5. **Redeem-disguised-as-refund**: Watcher `isRedeemTx()` checks if "refund" actually pays destination recipient.
6. **After COBI refund**: Executor should instant-refund user's source HTLC.

### General Debugging
- **Loki queries**: Filter by `solver_id` + `service_name` (alpen-executor, alpen-watcher)
- **Time window**: `created_at` to +1hr
- **Health check**: Executor `:4527/health`, `:4527/accounts`
- **On-chain**: Verify HTLC address balance/txs via Electrs API
- **Timestamps**: `additional_data.alpen.initiate_detected_timestamp` vs `initiate_timestamp`
- **Ports**: 4428 (keystore unlocker), 4527 (HTTP server)
- **Cache**: LevelDB at `cache_batcher`, app cache at `cacheDir` — restart clears in-memory
