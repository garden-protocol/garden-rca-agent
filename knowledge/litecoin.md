# Litecoin Chain Knowledge Base

Generated by Claude Code from source. Covers 2 repos + shared libraries:
- **watcher** — `litecoin-watcher` (Go, branch: feat/ltcsuite) — monitors Litecoin HTLC events
- **executor** — `litecoin-executor` (Go, branch: staging) — executes swap actions on Litecoin
- **shared** — `blockchain` (Go, branch: feat/litecoin) — HTLC scripts, Taproot wallet, indexer client, batcher

**Key Insight**: Litecoin has NO separate relayer repo. The litecoin-executor handles execution and the litecoin-watcher (a near-clone of bitcoin-watcher using `ltcsuite/ltcd` instead of `btcsuite/btcd`) monitors on-chain events and writes state to the shared PostgreSQL database.

## 1. Service Architecture Overview

### What Each Service Does

**litecoin-executor** (Go 1.24)
Self-contained executor that polls the orderbook API for pending orders every 2 seconds, maps orders to HTLC actions (Init, Redeem, Refund, InstantRefund) via `OrderToActionMapper`, and executes them on-chain using the shared `blockchain/btc` library. Uses LevelDB for action caching and a `BatcherWallet` with RBF strategy for transaction building. The executor also handles EVM chains in its config (hybrid executor), but the Litecoin-specific code path uses `BitcoinHTLCActionExecutor` with `ltcsuite/ltcd` libraries.

**Key difference from Bitcoin**: This is a **fork of the Bitcoin executor (cobi-v2)** adapted for Litecoin. Module name is still `cobi/v2`. All `btcsuite/btcd` imports are replaced with `ltcsuite/ltcd`. No separate watcher/relayer — the executor is self-contained.

### Dependencies
- `github.com/ltcsuite/ltcd` — Litecoin-native fork of btcd (chaincfg, txscript, wire, ltcutil, btcec/v2/schnorr)
- `github.com/catalogfi/blockchain/btc` (replaced to `version.btcfi.wtf/hashiraio/blockchain`) — HTLC wallet, scripts, indexer client, batcher
- Vendored `third_party/ltcsuite-secp256k1/` — CGo bindings for Litecoin's secp256k1 with MWEB extensions

### Order Lifecycle on Litecoin

```
Orderbook API provides pending orders → Executor polls every 2s →
OrderMapper determines action (Init/Redeem/Refund/InstantRefund/NoOp) →
BitcoinHTLCActionExecutor filters already-initiated → Batch execution (or individual fallback) →
Transaction broadcast via Electrs API → Orderbook API detects on-chain events → State updated
```

## 2. Executor Flow

### Key Files
- `cmd/main.go` — entry point, wallet init, HTTP server startup
- `executor/executor.go` — main execution loop (2s ticker), order processing pipeline
- `executor/mapper.go` — order-to-action mapping logic (the brain)
- `executor/validator.go` — external order validation API client
- `bitcoin/actions.go` — Bitcoin/Litecoin HTLC action executor, chain params
- `server/server.go` — HTTP validation server, source init detection deadline check

### Startup Sequence
1. Load config from JSON file (default `config.json`)
2. Await keystore password via HTTP unlocker on port **4428**
3. Parse config, decrypt private keys
4. Initialize EVM chain action executors (for any EVM chains in config)
5. Initialize Bitcoin/Litecoin HTLC action executors via `BitcoinHTLCActionExecutor()`
6. Create main `Executor` with orderbook provider, all action executors
7. Start HTTP server on port **4527** (validation + account endpoints)
8. Register executor with order validator API
9. Start executor polling loop

### Main Execution Loop (`executor.go`)
Two tickers:
- **2-second**: `executeOrders(ctx)` — the main loop
- **1-hour**: cache cleanup (24h TTL)

`executeOrders()` pipeline:
1. Fetch pending orders from orderbook API (by filler IDs / solver pubkeys)
2. Filter whitelisted initiators (if set)
3. Get latest block numbers for refund-eligible chains
4. Remove duplicates and blacklisted orders
5. Map orders to HTLC actions via `OrderToActionMapper`
6. Filter already-executed actions (via LevelDB cache)
7. Split by chain name
8. Execute each chain's actions concurrently (goroutines + WaitGroup)

### Order-to-Action Mapping (`mapper.go`)

| Condition | Action |
|-----------|--------|
| `destination_swap.initiate_tx_hash == ""` | **Init** (if confirmations ok, amount ok, deadline ok, price ok) |
| `source_swap.redeem_tx_hash == ""` AND `destination_swap.secret != ""` | **Redeem** (source HTLC with secret) |
| `destination_swap.refund_tx_hash == ""` AND `destination_swap.initiate_tx_hash != ""` | **Refund** (dest HTLC after timelock) |
| `destination_swap.refund_block_number > 0` AND no redeem/refund on source | **Instant Refund** (source HTLC via SACP) |
| deadline passed + 30min buffer | **Instant Refund** instead of Init |
| Otherwise | **NoOp** |

#### Init Gating Logic (`handleInit`):
1. Source confirmations >= required confirmations
2. Filled amount == order amount
3. Deadline not passed (configurable via `ignoreOrderDeadline`, default 12 hours from quote)
4. Price hasn't dropped beyond threshold (default 1%, configurable via `priceDropThreshold`)
5. External validation via `OrderValidator.Validate()` (calls HTTP API at `/{chain}/validate`)

#### Source Init Detection Deadline (`server.go:checkSrcInitDetectionDeadline`):
Uses `additional_data.bitcoin.initiate_detected_timestamp` to verify source init was detected before the quote deadline. Falls back to `additional_data.deadline` if `src_init_detection_deadline` not set.

### Action Execution (`bitcoin/actions.go`)

`BitcoinHTLCActionExecutor.Execute()`:
1. Filter already-initiated HTLCs (calls `swapper.IsInitiated()` on-chain check)
2. Try batch execution first (all actions in one tx)
3. If batch fails, fall back to individual execution per action
4. Each action converted to `btc.RawHTLCAction` with `btc.HTLCAction` enum
5. Calls `swapper.Execute(ctx, swaps)`

### Chain Params Selection (`bitcoin/actions.go`)
`parseLitecoinChainParams()` maps chain asset strings:
- `"regtest"` → `chaincfg.RegressionNetParams`
- `"testnet"` → `chaincfg.TestNet4Params`
- `"litecoin"` → `chaincfg.MainNetParams`

## 3. Watcher Flow

### Key Files
- `watcher/watcher.go` — core watcher loop, job scheduling, event processing
- `watcher/error.go` — error type classification (NonRecoverable/Recoverable/Ignorable)
- `blockchain/btc/event.go` — HTLC event detection from indexer transactions
- `blockchain/btc/scripts.go` — Taproot script leaf validation (IsRedeemLeaf, IsRefundLeaf, IsMultiSigLeaf)
- `blockchain/btc/indexer.go` — Electrs indexer client (HTTP REST)
- `store/store.go` — DB operations (GORM + PostgreSQL)
- `model/model.go` — data models, chain constants
- `config/config.go` — watcher configuration
- `cache/cache.go` — Ristretto-based event cache and job dedup

**Module name**: `github.com/catalogfi/bit-ponder` (shared codebase name with bitcoin-watcher)
**Go module**: Go 1.24.9, uses `github.com/ltcsuite/ltcd` v0.23.6 (Litecoin-native fork of btcd)

### Startup (`cmd/local/local.go`)
1. Load config from JSON via vault-go (supports encrypted config)
2. Connect to PostgreSQL via GORM
3. Create `ElectrsIndexerClient` pointing at Litecoin Electrs instance
4. Create `Watcher` with store, config, logger, indexer
5. `watcher.Start(ctx)` — launches workers + job scheduler + confirmation tracker
6. Start health check HTTP server on `:7288` (or `SERVER_PORT` env)

### Two Polling Tickers
1. **Main ticker** (configurable `interval` ms, default 5000ms): calls `scheduleJobs()` → `store.Observables()` → queues `Job` to `jobQueue` channel (buffered at 500)
2. **Confirmations ticker** (hardcoded 5s): calls `UpdateConfirmations()` → gets tip block height → updates `current_confirmations` for all swaps where `required > current`

### Worker Pool
N worker goroutines (configurable, default 4) consume from `jobQueue`. Each worker:
- 5-second context timeout per job
- Calls `processObservable(ctx, observable)`
- On error: logs `"Error processing observable"` with address

### processObservable — The Core Loop
For each observable HTLC address:
1. **Blacklist check** → skip if in `blacklistCache`
2. **Address decode**: `ltcutil.DecodeAddress(address, chainCfg)`
3. **Fetch HTLC events**: `htlcClient.HTLCEvents(ctx, asset, 0, 0)` via Electrs
4. **Remove duplicate events** (known bug workaround for partial inits)
5. **Event cache check** → skip if unchanged
6. **Filter events** into: HTLCInitiated, HTLCRedeemed, HTLCRefunded
7. **Handle inits** (`handleInits`): screen VIN addresses for blacklisting, aggregate filled amount, set initiate tx hash + block
8. **Handle redeems** (`handleRedeems`): extract secret, set redeem tx hash + block
9. **Handle refunds** (`handleRefundOrRedeem`): distinguish refunds from redeems-disguised-as-refunds (checks if output goes to `destination_recipient`)
10. **Timestamps**: set `initiate_timestamp`, `redeem_timestamp`, `refund_timestamp` + detection timestamps in `additional_data`
11. **Blacklist timing check**: if `initiate_timestamp < created_at` → blacklist
12. **DB write**: `store.UpdateSwap(swap, integratorProvider)`
13. **Cache events**

### HTLC Event Detection from Chain
`HTLCEvents()` queries Electrs (`/address/{addr}/txs`) and classifies:
- **VOUTs to HTLC address** = `HTLCInitiated` (amount, block height)
- **VINs from HTLC address** = Redeem or Refund, determined by Taproot witness parsing:
  - `IsRedeemLeaf(script)` → redeem (secret = `witness[1]`)
  - `IsRefundLeaf(script)` → refund (timelock path)
  - `IsMultiSigLeaf(script)` → instant refund (2-of-2 multisig)
  - Also supports legacy SegWit v0 HTLCs (`IsHtlc(script)` with branch byte detection)

### Observable Query
`store.Observables()` fetches swaps where:
- `chain = litecoin_* AND asset = primary AND amount > 0`
- Created within last 7 days
- Either: no initiate_tx_hash (within `order_deadline`), OR has initiate but no redeem/refund block numbers

**Notable**: `Observable.Address()` returns `o.Swap.SwapID` — the SwapID IS the HTLC Taproot address.

### Watcher Configuration
```json
{
  "mode": {"mode": "mainnet|testnet|localnet"},
  "database": {"db_url": "postgres://..."},
  "watcher": {"interval": 5000, "workers": 4, "cache_ttl": 172800},
  "litecoin": {"rpc": "http://litecoin-electrs:50001", "chain": "litecoin", "asset": "primary"},
  "screener_url": "...",
  "integrator_info": {"api_url": "", "allowed_mismatch_percent": 3},
  "order_deadline": 3600
}
```

### Indexer (Electrs-compatible API)
Both watcher and executor use an Electrs/Esplora-style REST API:
- `GET /address/{addr}/utxo` — Get UTXOs
- `GET /address/{addr}/txs` — Get tx history
- `GET /blocks/tip/height` — Get tip block height
- `GET /tx/{txid}` — Get tx details
- `GET /tx/{txid}/hex` — Get raw tx hex
- `POST /tx` — Submit raw tx
- `GET /fee-estimates` — Fee estimation

## 4. HTLC Interaction

### HTLC Type: Tapscript V2 (identical to Bitcoin)

The Litecoin HTLC uses **Taproot/Tapscript** with three leaf scripts in a Taproot script tree:

**Leaf 1 — Redeem:**
```
OP_SHA256 <secret_hash> OP_EQUALVERIFY <redeemer_xonly_pubkey> OP_CHECKSIG
```

**Leaf 2 — Refund:**
```
<timelock> OP_CHECKSEQUENCEVERIFY OP_DROP <initiator_xonly_pubkey> OP_CHECKSIG
```

**Leaf 3 — Instant Refund (2-of-2 MultiSig):**
```
<initiator_pubkey> OP_CHECKSIG <redeemer_pubkey> OP_CHECKSIGADD OP_2 OP_NUMEQUAL
```

### Internal Key (NUMS Point)
Same as Bitcoin: BIP-341 H value + `sha256("GardenHTLC")` as scalar `r`. NUMS = H + r*G. Key-path spend is unusable.

### Key Files
- `blockchain/btc/scripts.go` — `RedeemLeaf()`, `RefundLeaf()`, `MultiSigLeaf()`, `HtlcScriptV2()`
- `blockchain/btc/htlc.go` — `HTLCWallet`: `Initiate()`, `Redeem()`, `Refund()`, `Execute()`, `IsInitiated()`, `GenerateInstantRefundSACP()`
- `blockchain/btc/batcher.go` — Batches multiple HTLC actions into single transactions
- `blockchain/btc/indexer.go` — Electrs REST API client

### Instant Refund Flow (SACP)
1. User's wallet pre-signs a SACP tx with `SigHashSingle|AnyoneCanPay`
2. SACP tx bytes stored in `additional_data.instant_refund_tx_bytes`
3. Executor validates SACP tx (correct inputs, witness, control block)
4. Counter-signs at designated witness index (0 or 1)
5. Broadcasts completed 2-of-2 signed tx

### Wallet & Transaction Building
- **BatcherWallet** with RBF strategy (`btc.WithStrategy(btc.RBF)`)
- PTI (Pending Transaction Interval) of 2 seconds
- LevelDB cache for batcher state (`cache_batcher`)

## 5. Differences from Bitcoin

| Aspect | Bitcoin (cobi-v2) | Litecoin (litecoin-executor) |
|--------|-------------------|------------------------------|
| Go module | `btcsuite/btcd` | `ltcsuite/ltcd` |
| secp256k1 | btcsuite/btcd/btcec | ltcsuite/ltcd/btcec + vendored `third_party/ltcsuite-secp256k1` |
| Network params | Bitcoin `MainNetParams` | Litecoin `MainNetParams` (different prefixes, block times) |
| Address format | `bc1p...` (Bech32m) | `ltc1p...` (Bech32m) |
| Chain detection | `strings.Contains(chainAsset, "bitcoin")` | `strings.Contains(chainAsset, "litecoin")` |
| Config method | `TryBitcoin()` | `TryLTC()` |
| Swap detection | `IsBitcoinSwap()` | `IsLiteCoinSwap()` |
| Block time | ~10 min | ~2.5 min |
| HTLC script | Identical Tapscript V2 | **Identical** |
| Separate watcher? | Yes (bitcoin-watcher) | **No** — relies on orderbook API |
| Separate relayer? | Yes (btc-relayer) | **No** — bundled in executor |
| Ports | 4428 (unlocker), 4527 (server) | Same |

## 6. Error Patterns

### Watcher Errors
| Error Pattern | Source | Meaning |
|---|---|---|
| `"Error processing observable"` | `watcher/watcher.go` | Worker failed on a job |
| `"Failed to fetch and queue jobs"` | `watcher/watcher.go` | DB observables query failed |
| `"Failed to update current confirmations"` | `watcher/watcher.go` | Block height fetch or DB update failed |
| `"failed to decode address %s"` | `watcher/watcher.go` | Invalid HTLC address format |
| `"failed to get HTLC events for %v"` | `watcher/watcher.go` | Electrs query failure |
| `"error handling init events for %s"` | `watcher/watcher.go` | Init processing failure |
| `"error handling refund events for %s"` | `watcher/watcher.go` | Refund processing failure |
| `"failed to update swap in database"` | `watcher/watcher.go` | DB write failure |
| `"init tx timestamp is before created_at"` | `watcher/watcher.go` | Temporal inconsistency → blacklisted |
| `"Blacklisted address found"` | `watcher/watcher.go` | Compliance screener flagged (WARN) |
| `"order not found for swap id %s"` | `watcher/watcher.go` | Missing matched order during refund check |

### Executor Errors
| Error Pattern | Source | Meaning |
|---|---|---|
| `"could not convert the swap into bitcoin compatible swap"` | `bitcoin/actions.go` | Initiator/redeemer/secretHash not 64 chars or amount is 0 |
| `"error checking if HTLC is initiated with secret hash"` | `bitcoin/actions.go` | Indexer/network error checking HTLC status |
| `"BitcoinActionExecutor: invalid action"` | `bitcoin/actions.go` | Unknown action type |
| `"BitcoinActionExecutor: no response"` | `bitcoin/actions.go` | Empty response from individual execution |
| `"No chain worker found for chainAsset"` | `executor/executor.go` | Config missing for chain |
| `"failed to get pending orders"` | `executor/executor.go` | Orderbook API unreachable |
| `"need more %d blocks to refund"` | `blockchain/btc/htlc.go` | Timelock not yet expired |
| `"invalid signed input"` / `"invalid output script"` | `blockchain/btc/htlc.go` | SACP tx validation failed |
| `"invalid secret"` | `blockchain/btc/htlc.go` | Secret hash mismatch during redeem |
| `"invalid chain asset"` | `bitcoin/actions.go` | Unrecognized chain in asset string |
| `"insufficient balance"` | `blockchain/btc/indexer.go` | Wallet doesn't have enough LTC |
| `"source initiate not yet detected by indexer"` | `server/server.go` | Init detection timestamp missing |
| `"source initiate detected after deadline"` | `server/server.go` | Init detected too late |
| `"script address mismatch"` | `server/server.go` | Computed HTLC address doesn't match swap ID |
| `"insufficient confirmations"` | `server/server.go` | Not enough block confirmations |
| `"source swap validation failed"` | `executor/mapper.go` | External validator API rejected |

### Indexer Error Codes
| Error String | Meaning |
|---|---|
| `"transaction already in block chain"` | Tx already confirmed |
| `"bad-txns-inputs-missingorspent"` | UTXOs already spent |
| `"txn-mempool-conflict"` | Conflicting tx in mempool |

## 7. Investigation Playbook

### missed_init (Destination initiate not happening)

1. **Check executor logs** for the order's `createID` or `secretHash`:
   - `"Mapper: NoOpHTLCAction"` → mapper decided not to act
   - `"source swap validation failed"` → external validator rejected
2. **Check confirmations**: Source `current_confirmations >= required_confirmations`?
3. **Check deadline**: `time.Now() > deadline + (ignoreOrderDeadline - 1) hours`? Default window = deadline + 11 hours. After that + 30 min, triggers instant refund instead.
4. **Check price protection**: Price dropped > `priceDropThreshold` (default 1%)? Look for NoOp after price check.
5. **Check filled amount**: Source `filled_amount == amount`? Partial fill → executor waits.
6. **Check source init detection deadline**: `initiate_detected_timestamp` after deadline → order rejected by validator.
7. **Check `IsInitiated` filter**: Pre-check if dest HTLC already has funds. Could be false positive if indexer stale.
8. **Check wallet balance**: Look for `"insufficient balance"` errors.
9. **Check indexer**: `"failed to get pending orders"` or timeout from Electrs API.

### stuck_order (Not progressing)

1. **Identify stuck state**: Which swap fields are populated?
   - Source inited, dest not → see missed_init
   - Dest inited, source not redeemed → secret not extracted yet? Check `destination_swap.secret`
   - Both initiated, secret available → check for execution errors
2. **Check cache**: LevelDB cache (`cache_batcher` + app cache). Action IDs = `{secretHash}_{actionType}`. If cached, won't retry.
3. **Check batch failures**: Batch fails → individual fallback → individual fails → error logged per action.
4. **Check blacklist**: `additional_data.is_blacklisted == true` → filtered out.

### refunded

1. **Identify which side**: `destination_swap.refund_tx_hash` (COBI refunded) or `source_swap.refund_tx_hash` (user refunded).
2. **Destination refund**: Check `block_number >= initiate_block_number + timelock`.
3. **Instant refund**: Check `additional_data.instant_refund_tx_bytes`. If empty for Litecoin swap, instant refund is skipped.
4. **SACP validation errors**: Look for SACP validation failures in logs.
5. **Timeline**: Instant refund after deadline only triggered after `deadline + 30 minutes` buffer.

### General Debugging
- **Loki queries**: Filter by `solver_id` + `service_name` (litecoin-executor), window = `created_at` to +1hr
- **Env vars**: `WEBHOOK_URL` (Discord alerts), `ignoreOrderDeadline` (hours), `priceDropThreshold` (float)
- **Ports**: 4428 (keystore unlocker), 4527 (HTTP validation + accounts)
- **Cache**: LevelDB at `cache_batcher` dir, app cache at `cacheDir` — restart clears in-memory cache
- **Indexer health**: Electrs API at configured `rpcUrl` for the chain
