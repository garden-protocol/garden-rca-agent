# Starknet Chain Knowledge Base

Generated by Claude Code from source. Covers 3 repos:
- **watcher** — `garden-starknet-watcher` (Rust, branch: staging)
- **executor** — `starknet-executor` (Rust, branch: staging)
- **relayer** — `starknet-relayer` (Rust, branch: staging)

Shared library: `garden-rs` — orderbook, starknet HTLC/multicall/ERC20 wrappers, primitives, fiat, auth

---

## 1. Service Architecture Overview

### What Each Service Does

**garden-starknet-watcher** (Rust, async/tokio)
Monitors Starknet HTLC contracts for Initiated, Redeemed, and Refunded events. Writes state changes to the shared PostgreSQL orderbook database. Uses `starknet-rs` v0.17.0 for RPC communication. Runs N worker goroutines that consume from an EventQueue. Employs `moka` cache (TTL=500s, max=10000) for deduplicating already-processed events.

**starknet-executor** (Rust, async/tokio, axum)
Solver-side automated execution engine. Polls pending orders, maps them to HTLC actions (InitiateWithSignature, Redeem, Refund, InstantRefund) via `OrderMapper`, simulates via dry-run, submits batched multicall transactions on-chain. Uses a keystore with HTTP unlocker (port 4529). Runs HTTP server on port 4532 for health/validate/accounts endpoints. Registers with executor registry on startup.

**starknet-relayer** (Rust, async/tokio, axum)
User-facing relay service. Exposes HTTP API on port 4436 for user-initiated operations (initiate with signature, redeem). Also runs a background `RedeemerService` that auto-redeems destination swaps when secrets become available. Uses channel-based architecture with SwapTransmitter → Executor pipeline. Batches up to 200 swaps per multicall.

### Order Lifecycle on Starknet

```
User signs SNIP-12 typed data → Relayer /initiate endpoint → SwapTransmitter validates (dry-run) →
Executor batches into multicall → HTLC initiated on-chain → Watcher detects Initiated event →
DB updated → Credentials provider stores secret → RedeemerService polls → fetches secret →
Submits redeem via multicall → HTLC redeemed on-chain → Watcher detects Redeemed event → DB updated
```

## 2. Watcher Flow

### Key Files
- `src/watcher.rs` — main polling loop
- `src/watcher/handlers.rs` — event handlers (initiated/redeemed/refunded)
- `src/watcher/worker.rs` — worker goroutines consuming from EventQueue
- `src/watcher/block_provider.rs` — background block number updater (every 500ms)
- `src/watcher/event_queue.rs` — bounded event queue
- `src/primitives.rs` — event parsing from Starknet events

### Startup
1. Loads config from Vault (`vault-rs`), sets up Discord webhook tracing
2. `setup_watchers()` creates one `Watcher` per HTLC contract in `settings.htlc_contracts`
3. Each watcher gets: JsonRpcClient, OrderbookProvider (postgres), EventQueue, BlockNumberProvider, moka Cache

### Event Polling Loop
1. Spawns N workers (`settings.worker_count`) consuming from EventQueue
2. Infinite loop: `poll_events()` → sleep `polling_interval` ms
3. `poll_events()`: gets `latest_block`, computes `to_block = min(current + max_block_span, latest)`
4. Calls `process_all_events(current, to_block)` via `starknet_client.get_events(filter, continuation_token, 100)` with pagination
5. Filters already-processed events via moka cache (key = txHash + event keys)
6. Pushes new events to EventQueue
7. Calls `orderbook.update_confirmations(blockchain, latest_block)` to update confirmation counts

### Event Parsing (Starknet Selectors)
- `selector!("Initiated")` → keys[1]=order_id, data[0..2]=secret_hash (two u128s), data[2..]=amount (U256)
- `selector!("Redeemed")` → keys[1]=order_id, data[0..2]=secret_hash, data[2..10]=secret (eight u32s)
- `selector!("Refunded")` → keys[1]=order_id

### Worker Processing
Workers dispatch to handlers:
- `handle_initiated()` → `orderbook.update_swap_initiate(order_id, filled_amount, tx_hash, block, timestamp, asset_id)`
- `handle_redeemed()` → `orderbook.update_swap_redeem(order_id, tx_hash, secret, block, timestamp, asset_id)`
- `handle_refunded()` → `orderbook.update_swap_refund(order_id, tx_hash, block, timestamp, asset_id)`

Error handling: "Swap not found" → event dropped silently. Other errors → event re-pushed to queue for retry.

### Configuration
```toml

rpc_url = "http://starknet-rpc/rpc"
start_block = 0
max_block_span = 1000
polling_interval = 100
blockchain = "starknet"
worker_count = 1
queue_size = 1000
htlc_contracts = ["0x..."]
```

## 3. Executor Flow

### Key Files
- `src/executor.rs` — main execution loop
- `src/htlc.rs` — HTLC calldata generation (GardenHTLC)
- `src/main.rs` — startup, keystore, wallet init
- `src/server/handlers.rs` — HTTP validation endpoints
- `src/validator.rs` — executor registry validation
- `src/register.rs` — executor registration

### Startup
1. Reads config from TOML, awaits keystore password via HTTP on port 4529
2. Creates `SingleOwnerAccount` with `ExecutionEncoding::New`, block set to `PreConfirmed`
3. Initializes `MulticallContract` with NonceCounter
4. Creates `FiatProvider` and `OrderMapper` with `price_drop_threshold`
5. Creates `GardenHTLC` wrapper, `ValidatorService`
6. Starts HTTP server on port 4532 (health, validate, accounts)
7. Registers with executor registry
8. Enters main execution loop

### Main Execution Loop
1. `execute_pending_swaps()` every `polling_interval` ms (default 5000ms)
2. Fetches pending swaps from PendingOrdersProvider filtered by `chain_name` and `executer_address`
3. For each order, `order_mapper.map()` determines action
4. For Initiate actions, validates via `validator.is_valid(chain, SourceInitiate, order)`
5. Maps Garden actions to Starknet actions: **Initiate → InitiateWithSignature** (multicall requires signature variant)
6. Filters cached swaps (moka cache, TTL=30min)
7. Groups by asset, executes multicall per asset group

### Action Execution
1. `htlc.get_multicall_calldata(asset, requests)` builds batch calldata
2. `multicall_contract.multicall(call_data)` submits
3. On success: caches all swap keys, logs tx hash
4. On failure: resets nonce, does dry-run filtering (`filter_with_dry_run`) to identify failing individual requests, retries with filtered set

### HTLC Calldata Generation (GardenHTLC)
Four action types:
- **InitiateWithSignature**: Signs SNIP-12 typed data with executor key, builds `initiate_with_sig_calldata`
- **Redeem**: Converts secret hex to u32[8] array, builds `redeem_calldata`
- **Refund**: Builds `refund_calldata`
- **InstantRefund**: Signs SNIP-12 typed data, builds `instant_refund_calldata`

HTLC address lookup is cached (moka). On first access, checks/sets ERC20 allowance (max U256).

### On-Chain Validation
`/validate` endpoint reads on-chain state via `get_order` (calls `selector!("get_order")` on HTLC contract) and validates amount, timelock, initiator, redeemer, initiated_at against orderbook.

## 4. Relayer Flow

### Key Files
- `src/main.rs` — startup, wallet, spawns services
- `src/server/handlers.rs` — HTTP endpoints (initiate, redeem, get_action_calldata)
- `src/executor.rs` — SwapTransmitter / Executer (channel-based batching)
- `src/redeemer/mod.rs` — RedeemerService (auto-redeem loop)
- `src/htlc.rs` — HTLC calldata helpers
- `src/calldata.rs` — calldata generation for client-side signing

### API Endpoints

**POST /initiate** (`initiate_with_sig_handler`):
1. Cache check (prevents duplicates, TTL=500s)
2. Fetches swap from orderbook DB
3. Validates swap not already initiated
4. Creates Order with `HTLCAction::InitiateWithUserSignature { signature }`
5. Sends to SwapTransmitter → validates (dry run) → queues for executor

**POST /redeem** (`redeem_handler`):
1. Cache check
2. Fetches swap, validates initiated but not redeemed
3. Validates secret: `sha256(secret_bytes) == swap.secret_hash`
4. Creates Order with `HTLCAction::Redeem { secret }`
5. Sends to SwapTransmitter

**PATCH /:version/:order_id?action=initiate|redeem** — Unified v2 endpoint.

**GET /:version/:order_id?action=initiate** — Returns `InitiateParams { approval_transaction, initiate_transaction, typed_data }` for client-side SNIP-12 signing. Checks ERC20 allowance.

### SwapTransmitter / Executor
Channel-based (bounded channel, cap=100):
1. `SwapTransmitter.send(order)` validates via dry run, sends to channel
2. `Executer.run()` collects up to BATCH_SIZE=200 swaps, groups by asset
3. Builds multicall per asset group
4. Spawns background task to wait for L2 acceptance (600 retries, 5s interval = ~50 min)
5. On revert: resets nonce. On success: sends tx hash via response channels
6. Deduplicates by swap_id

### RedeemerService
Auto-redeem background loop polling every 2s:
1. Fetches pending orders from PendingOrdersProvider
2. Filters: source initiated + destination initiated + not redeemed + not refunded + destination chain matches
3. Fetches credentials (secrets) from OrderCredentialsProvider in batch
4. Validates secret hash, submits redeem through SwapTransmitter
5. Caches processed + non-redeemable orders

### Configuration
```toml

rpc_url = "http://starknet-rpc/rpc"
```

## 5. HTLC Contract Interaction

### Contract Type
Cairo smart contracts on Starknet. Uses SNIP-12 typed data for signatures (Starknet equivalent of EIP-712).

### Contract Functions
| Function | Parameters | Returns | Purpose |
|----------|-----------|---------|---------|
| `get_order(order_id)` | Felt | 7 Felts: [is_fulfilled, initiator, redeemer, initiated_at, timelock, amount_low, amount_high] | Read on-chain HTLC state |
| `initiate(redeemer, timelock, amount, secret_hash)` | | | Standard initiate |
| `initiate_with_signature(initiator, redeemer, timelock, amount, secret_hash, signature)` | | | Third-party initiate with SNIP-12 sig |
| `redeem(order_id, secret)` | secret = 8 u32 values | | Claim funds with secret |
| `refund(order_id)` | | | Timelock-based refund |
| `instant_refund(order_id, signature)` | | | Executor-signed instant refund |
| `token()` | | Felt | ERC20 token address for this HTLC |

### Events
| Event | Keys | Data |
|-------|------|------|
| `Initiated` | [selector, order_id] | [secret_hash_hi, secret_hash_lo, amount_lo, amount_hi] |
| `Redeemed` | [selector, order_id] | [secret_hash_hi, secret_hash_lo, secret[0..8]] |
| `Refunded` | [selector, order_id] | |

### Multicall Pattern
Both executor and relayer use `MulticallContract` to batch calls:
- Calldata: `[target_address, calls_len, (call_len, selector, calldata...)...]`
- Uses NonceCounter for sequential nonce management
- On failure, nonce is reset via `reset_nonce()`

### SNIP-12 Typed Data (Signature Types)
- **Initiate**: `{redeemer, amount{low,high}, timelock, secretHash[2], verifyingContract}` domain: HTLC v50
- **InstantRefund**: `{orderID, verifyingContract}` domain: HTLC v50

## 6. Error Patterns

### Watcher Errors
| Error Message | Meaning | Severity |
|---|---|---|
| `"Error fetching events: {}"` | RPC get_events failed | ERROR — retries next poll |
| `"Error processing events: {}"` | Block processing failed | ERROR — sleeps and retries |
| `"Error updating confirmations: {}"` | DB update failed | ERROR |
| `"block number not found in event"` | Event missing block number | ERROR — skips event |
| `"pending block with events found"` | Pre-confirmed block encountered | ERROR — skips block |
| `"Error parsing event: {}"` | Malformed event data | ERROR — continues |
| `"Failed to update initiate/redeem/refund"` | DB write failed | ERROR — re-queued |
| `"Swap not found"` (in error) | Order not in DB yet | Dropped silently |
| `"Latest block is behind current block"` | RPC node lagging | WARN |

### Executor Errors
| Error Message | Meaning |
|---|---|
| `"Failed to parse action for order"` | OrderMapper could not determine action |
| `"Order is not valid"` | On-chain validation failed (amount/timelock/addresses mismatch) |
| `"Non-NoOp action missing swap data"` | Swap data missing for actionable order |
| `"Unsupported action by executor"` | Action type not handled |
| `"Failed to execute multicall transaction"` | On-chain submission failed |
| `"Failed to reset nonce"` | Post-failure nonce reset failed (critical) |
| `"Failed to dry run request"` | Individual action simulation failed |
| `"Error in execution cycle: {}"` | Top-level loop error |
| `"Failed to create wallet"` | Keystore decryption failed |
| `"Failed to register executor"` | Registry communication failed |
| `"Allowance confirmation timed out after 60 seconds"` | ERC20 approve tx not confirmed |

### Relayer Errors
| Error Message | Meaning |
|---|---|
| `"Order already initiated"` | Cache hit or DB shows initiate_tx_hash |
| `"Order not found"` | Order not in orderbook DB |
| `"Order not initiated"` | Trying to redeem before initiation |
| `"Order already redeemed"` | redeem_tx_hash already set |
| `"Invalid secret"` | SHA256(secret) != secret_hash |
| `"swap queue is full"` | Channel buffer (100) exhausted |
| `"timeout"` | 30-second tx confirmation timeout |
| `"Transaction reverted: ... reason: ..."` | On-chain tx reverted |
| `"Max retries reached for tx"` | 600 retries (50 min) exhausted |
| `"Asset not found in htlc_wallets"` | Unknown HTLC contract address |
| `"Secret hash mismatch"` (redeemer) | Credential secret doesn't match order |

## 7. Investigation Playbook

### missed_init (Destination initiation not happening)

1. **Check executor logs** for the order:
   - `"Fetched pending orders"` — is the order in the pending list?
   - `"Mapped action for order"` — what action was mapped?
   - `"Order is not valid"` — on-chain validation failed
   - `"Failed to parse action for order"` — OrderMapper error, possibly price drop
2. **Check confirmations**: Source swap must have sufficient confirmations
3. **Check ERC20 allowance**: Look for `"Approving allowance"` and `"Allowance confirmation timed out"`
4. **Check multicall execution**: `"Failed to execute multicall transaction"` → dry-run filter step
5. **Check nonce**: `"Failed to reset nonce"` → all subsequent txs fail, needs restart
6. **Check on-chain**: Call `get_order(swap_id)` to verify if initiated on-chain but watcher missed it
7. **Check executor wallet**: Sufficient token balance for initiation?

### stuck_order (UserRedeemPending)

1. **Check relayer RedeemerService**: `"Received pending orders"` and `"Found redemption candidates"`
2. **Filter conditions**: source initiated (block > 0) + destination initiated (block > 0) + not redeemed + not refunded + destination chain matches
3. **Check credentials**: secret fetch from order_credentials_url. Missing secret → `non_redeemable_cache` for 1 hour
4. **Check for errors**: `"Secret hash mismatch"`, `"swap queue is full"`, `"timeout"`
5. **Check watcher**: Is it stuck? `"Latest block is behind"`, `"Error fetching events"`
6. **Verify HTLC contract address**: Is it in `htlc_contracts` config?

### stuck_order (SolverRedeemPending)

1. **Check executor**: Is it polling? Does it see the order as pending?
2. **Check secret availability**: Destination must be redeemed to reveal secret
3. **Check action mapping**: Should be `Redeem` action
4. **Check multicall**: Execution or dry-run failures?
5. **Check nonce state**: Stuck nonce blocks all txs

### refunded

1. **Check watcher logs**: `"Refunded"` event with order_id and tx_hash
2. **Determine type**: Timelock (initiated_at + timelock < current_time) vs instant refund
3. **For instant refund**: Check executor logs for `"Generated calldata for InstantRefund"`
4. **Check timeline**: created_at → source_init → dest_init → (expected redeem but missed) → refund
5. **Check if both sides refunded**: Source and destination refund status

### General Debugging

- **Loki queries**: Filter by `solver_id` + `service_name` (starknet-executor, starknet-relayer, starknet-watcher)
- **Time window**: `created_at` to `created_at + 1hr`
- **Health checks**: Relayer `/health`, Executor `/health`
- **Accounts**: Executor `/accounts` returns chain name and executor address
- **On-chain state**: Call `get_order(swap_id)` on HTLC contract
- **Nonce issues**: Look for `"Failed to reset nonce"` — service restart may be needed
- **Cache effects**: Executor 30min, Relayer 500s, Watcher 500s — restart clears caches
